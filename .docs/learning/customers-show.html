顧客詳細
顧客の基本情報と受注履歴を表示する画面
URL /customers/{id}
HTTPメソッド GET
アクセス権限 sales accounting admin
画面項目 — 顧客基本情報
項目名 説明 値なしの場合 表示条件
会社名 顧客の法人名。ページのヘッダーとして表示 — 全ロール
担当者名 顧客側の窓口担当者 「—」と表示 全ロール
電話番号 連絡先電話番号 「—」と表示 全ロール
メールアドレス 連絡先メールアドレス 「—」と表示 全ロール
与信枠 取引可能な上限金額（円）。カンマ区切りで表示 0 全ロール
住所 顧客の所在地 「—」と表示 全ロール
編集ボタン 顧客編集画面へ遷移 — sales / admin のみ
削除ボタン 確認ダイアログ表示後にDELETEリクエストを送信 — admin のみ
画面項目 — 受注履歴
項目名 説明 値なしの場合
受注番号 ORD-{年度}-{連番4桁} 形式の受注識別番号 「受注履歴はありません」と表示
受注日 受注確定日（confirmed_at）。なければ作成日（created_at）
ステータス 受注のステータス（確定済み / キャンセル 等）
操作 受注詳細へのリンク（受注詳細画面は別タスクで実装予定）
処理内容
  • 顧客情報の取得
    URLの {id} から対象の顧客レコードを取得する。存在しないIDの場合は404エラー。
  • 受注履歴の取得
    salesOrders リレーション（sales_orders.customer_id で結合）を id の降順で取得し、最新の受注から順番に表示する。
  • 削除処理
    削除ボタン押下 → 確認ダイアログ → DELETE送信。受注が1件でも存在する場合は CustomerHasOrdersException が発生し、削除を中止してエラーメッセージを表示する。
💡 なぜそうなっているのか
  • 受注履歴を顧客詳細に表示する理由: 「この顧客とどんな取引をしてきたか」を1画面で把握できるようにするため。営業担当者が顧客へ連絡する前に取引状況を素早く確認できる。
  • 受注が存在する顧客を削除できない理由: 顧客を削除すると、過去の受注データが「誰の注文か」を失ってしまう（外部キー制約の観点）。業務上も、取引履歴のある顧客を消すことは想定していない。
  • ソフトデリート（SoftDeletes）を使っている理由: 論理削除なのでDBに削除日時が記録される。誤削除してしまった場合でもDB管理者が復元可能。
