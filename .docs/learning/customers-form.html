顧客登録・編集
顧客の新規登録と既存顧客の情報更新を行う画面（共通フォーム）
URL（登録） /customers/create
URL（編集） /customers/{id}/edit
アクセス権限 sales admin
画面項目とバリデーション
項目名 必須 バリデーションルール なぜそのルールなのか
会社名 ● 文字列・最大255文字 顧客を識別する主たる名称のため必須。DBカラムの上限（VARCHAR 255）に合わせている
担当者名 任意 文字列・最大255文字 窓口担当者がいない・不明な場合もあるため任意
住所 任意 文字列・最大500文字 住所は番地・建物名など長くなりがちなため255より余裕を持たせている
電話番号 任意 文字列・最大20文字 数値型ではなく文字列型にしているのは、ハイフンあり（03-1234-5678）・なし両方を受け付けるため
メールアドレス 任意 メール形式・最大255文字 フォーマット検証により xxx@yyy.zzz 形式でない入力を弾いて入力ミスを防ぐ
与信枠 ● 整数・0以上 与信なし（0円）は許容するが、負の与信枠は業務上意味をなさないため0以上を必須とする
処理内容
  • バリデーション（サーバーサイド）
    StoreCustomerRequest / UpdateCustomerRequest のバリデーションルールに基づきチェック。エラーがあれば入力値を保持したまま（old()）フォームを再表示する。
  • 登録処理（新規登録の場合）
    CustomerService::create() を通じて customers テーブルに INSERT。成功後は顧客詳細画面（/customers/{id}）へリダイレクトし「顧客を登録しました」とフラッシュメッセージを表示。
  • 更新処理（編集の場合）
    CustomerService::update() を通じて対象レコードを UPDATE。成功後は顧客詳細画面へリダイレクトし「顧客情報を更新しました」とフラッシュメッセージを表示。
💡 なぜそうなっているのか
  • 登録後に詳細画面へリダイレクトする理由: 「今登録した内容を確認できる」UXのため。一覧へ戻すと、どのレコードが新しく追加されたか見つけにくい。
  • 登録と編集で同じフォームパーツ（_form.blade.php）を共用する理由: 同じ項目を2つのファイルに書くと、項目を追加・変更したときに両方直す手間が生じる（DRY原則）。
  • CustomerService を経由してDBに保存する理由: Controller → Service → Repository の3層構造で、ビジネスロジックをService層に集めることで、テストの書きやすさと保守性を高めるため。
