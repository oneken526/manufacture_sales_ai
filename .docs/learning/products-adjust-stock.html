在庫調整
棚卸しや入荷時に在庫数を手動で増減させる画面
URL（フォーム表示） GET /products/{id}/adjust-stock
URL（実行） POST /products/{id}/adjust-stock
アクセス権限 warehouse admin
画面項目 — 現在の在庫情報（参照）
項目名 説明
品番・製品名 調整対象製品の識別情報。フォームのヘッダーとして表示
実在庫数 現在の stock_quantity の値
引当中在庫数 現在の reserved_quantity の値（受注確定済み・出荷待ち分）
利用可能在庫数 stock_quantity - reserved_quantity の計算値
画面項目 — 入力フィールド
項目名 必須 バリデーションルール なぜそのルールなのか
増減数 ● 整数・0以外 プラスで増加、マイナスで減少。0の調整は意味がなくログだけが増えるため入力不可
調整理由（メモ） 任意 文字列・最大500文字 なぜ在庫を調整したかを記録するため。後日の監査・トレーサビリティに役立つ
処理内容
  • バリデーション
    AdjustStockRequest のルールで増減数（必須・整数・0以外）とメモ（任意・最大500文字）をチェック。
  • 整合性チェック
    調整後に stock_quantity + 増減数 < reserved_quantity となる場合は StockAdjustmentViolatesIntegrityException をスローしてエラーメッセージを表示する（在庫を引当数より少なくすることは禁止）。
  • 在庫更新
    products.stock_quantity を 増減数 分だけ加算（マイナス値の場合は減算）する。
  • 在庫変動履歴の記録
    stock_movements テーブルに reason = manual_adjustment（コード値: 5） として操作ログを自動記録する。操作したユーザーID・日時・増減数・メモがすべて保存される。
  • 二重送信防止
    「調整を実行する」ボタン押下後にJavaScriptでボタンをdisabled化し「処理中...」と表示する。在庫調整は取り消しが難しいため、誤って2回送信されないよう防止している。
  • 完了後のリダイレクト
    調整成功後は製品編集画面（/products/{id}/edit）へリダイレクトし「在庫数を調整しました」と表示する。
💡 なぜそうなっているのか
  • 在庫変動をログに残す理由: 「いつ・誰が・何個増減したか」を追跡できるようにするため。棚卸しの監査対応やヒューマンエラーの調査で証跡が必要。
  • 引当数を下回る減算を禁止する理由: 引当中の在庫は受注確定済みで出荷予定の分。それを減らすと「受注確定したのに在庫がない」という矛盾が生じ、出荷できなくなる。
  • warehouse と admin のみに絞った理由: 在庫の増減は倉庫担当の業務であり、営業担当が独自に在庫を調整すると実態と乖離するリスクがあるため。
