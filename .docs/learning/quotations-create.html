見積新規作成
顧客と製品明細を指定して見積書を作成する画面
URL（フォーム） GET /quotations/create
URL（保存） POST /quotations
アクセス権限 sales admin
画面項目とバリデーション
項目名 必須 バリデーションルール なぜそのルールなのか
顧客（プルダウン） ● customers テーブルに存在するIDであること 存在しない顧客IDが送信されても受け付けないようにサーバーサイドで検証する
明細（1行以上必須） ● 配列・最低1件 明細が0件の見積は意味をなさないため
明細 → 製品（プルダウン） ● products テーブルに存在するIDであること 存在しない製品を見積に含めることはできない
明細 → 数量 ● 整数・1以上 0個・マイナスの注文は受注として成立しないため
明細 → 単価 ● 整数・0以上 無償提供（0円）は許容するが負の単価は不可
有効期限 任意 日付形式 見積の有効期限。この日付を過ぎると受注確定不可になる。省略可
備考 任意 文字列・最大1000文字 顧客への伝達事項・特記事項を記載するフリーテキスト
JavaScript による動的機能
機能 動作 仕組み
明細行の動的追加 「明細行を追加」ボタンで行を増やせる JavaScriptで新しい入力行のHTMLを生成して挿入
明細行の削除 各行の「削除」ボタンで行を取り除く JavaScriptで該当の行要素を DOM から削除
製品選択時の単価自動入力 製品プルダウン変更時に単価フィールドへマスタ単価を自動入力 製品データを data-unit-price 属性に埋め込み、選択時にJSで読み取り
金額のリアルタイム計算 数量・単価の入力ごとに各行の金額と合計金額を更新 内部APIエンドポイント POST /api/internal/quotations/calculate をAJAX呼び出し
処理内容（保存時）
  • バリデーション
    StoreQuotationRequest のルールで全項目を検証。エラー時は入力値を保持したまま再表示する。
  • 見積番号の採番
    DBトランザクション内で document_sequences テーブルを参照し、QUO-{西暦年}-{連番4桁} 形式の見積番号を発行する（例: QUO-2026-0001）。トランザクションにより同時登録時の番号重複を防ぐ。
  • 見積・明細の保存
    quotations テーブルに見積ヘッダーを INSERT し、各明細行を quotation_items テーブルに INSERT する。ログインユーザーIDが created_by として記録される。
  • 完了後のリダイレクト
    保存成功後は作成した見積の詳細画面（/quotations/{id}）へリダイレクトし「見積を作成しました」と表示する。
💡 なぜそうなっているのか
  • 単価をマスタから自動入力するが手動変更可能にした理由: 顧客ごとの値引き交渉や特別価格に対応するため。製品マスタの標準単価を変えずに、この見積だけ別の単価を設定できる。
  • 金額計算をサーバー側API（/api/internal/quotations/calculate）で行う理由: 計算ロジックをJavaScriptだけに任せると、改ざんされた数値で見積が作られるリスクがある。サーバー側で計算することで信頼性を確保している。
  • 見積番号採番をトランザクション内で行う理由: 複数の営業担当が同時に見積を作成した場合でも、同じ番号が発行されないようにするため。
  • 見積作成者（created_by）を記録する理由: 受注確定時に「誰が作った見積か」を追跡するため。在庫引当ログにも作成者IDが記録される。
