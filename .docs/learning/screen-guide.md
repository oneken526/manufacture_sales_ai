## 1. 業務フロー全体図

```
顧客マスタ登録
    │
    ▼
見積作成 ──── 製品マスタ参照
    │
    ▼
見積詳細確認・PDF出力（顧客へ送付）
    │
    ▼
受注確定（在庫引当が自動実行される）
    │
    ▼
受注一覧（※別タスクで実装予定）
    │
    ▼
出荷・請求（※別タスクで実装予定）
```

**ポイント:**
- 見積→受注確定の瞬間に「在庫引当」が発生し、製品の `reserved_quantity`（引当中在庫数）が増加する
- 受注は必ず「見積」を経由して作成される（見積なしの直接受注入力は存在しない）

## 2. ロールとアクセス権限

システムには4種類のロールがあり、画面ごとにアクセスできるロールが異なる。

| ロール | 説明 | 主な担当業務 |
|--------|------|-------------|
| **admin**（システム管理者） | すべての機能を利用できる | ユーザー管理、マスタ管理全般 |
| **sales**（営業担当者） | 顧客・見積を操作できる | 顧客登録、見積作成・受注確定 |
| **warehouse**（在庫・出荷担当者） | 在庫を操作できる | 在庫調整、出荷処理 |
| **accounting**（管理職・経理担当） | 参照系と経理業務 | 顧客・製品の閲覧、請求書発行 |

**なぜロールで画面を分けるのか:**  
操作権限を職務に合わせることで、誤操作や不正操作を防ぐため。例えば在庫の削除・調整は倉庫担当だけが行い、営業担当が誤って在庫を書き換えられないようにしている。

## 3. 顧客マスタ管理

### 3-1. 顧客一覧 `/customers`

**アクセス権限:** sales / accounting / admin

#### 画面項目

| 項目名 | 説明 |
|--------|------|
| 検索ボックス | 会社名・担当者名・電話番号で部分一致検索 |
| 会社名 | 顧客の法人名 |
| 担当者名 | 顧客側の窓口担当者 |
| 電話番号 | 連絡先電話番号 |
| メール | 連絡先メールアドレス |
| 与信枠 | この顧客に対して取引可能な上限金額（円） |
| 詳細リンク | 顧客詳細画面へ遷移 |
| 編集リンク | 顧客編集画面へ遷移（sales / admin のみ表示） |
| 削除ボタン | 顧客を削除（admin のみ表示） |

#### 処理内容

1. **検索:** URLパラメータ `?q=キーワード` を受け取り、会社名・担当者名・電話番号に対して `LIKE '%キーワード%'` で絞り込む
2. **ページネーション:** 1ページあたり50件表示。次のページへのリンクは検索キーワードを引き継ぐ
3. **新規登録ボタン:** sales / admin ロールにのみ表示される

**なぜそうなっているのか:**
- 検索を部分一致にしているのは、会社名の正式名称が分からなくても素早く探せるようにするため
- ページネーションを50件にしているのはシステム要件（NFR-021）で定められたパフォーマンス基準のため
- 削除ボタンを admin のみにしているのは、誤って顧客データを消す事故を防ぐため

---

### 3-2. 顧客詳細 `/customers/{id}`

**アクセス権限:** sales / accounting / admin

#### 画面項目

| 項目名 | 説明 |
|--------|------|
| 会社名 | ヘッダーとして表示 |
| 担当者名 | 値がない場合は「—」と表示 |
| 電話番号 | 値がない場合は「—」と表示 |
| メールアドレス | 値がない場合は「—」と表示 |
| 与信枠 | カンマ区切りの金額表示 |
| 住所 | 値がない場合は「—」と表示 |
| **受注履歴テーブル** | この顧客に紐づく受注の一覧（受注番号・受注日・ステータス） |
| 編集ボタン | sales / admin のみ表示 |
| 削除ボタン | admin のみ表示 |

#### 処理内容

1. **受注履歴の表示:** `salesOrders` リレーションを `id` の降順で取得し、最新の受注から順番に表示する
2. **削除処理:** 削除ボタン押下時に確認ダイアログを表示。受注が存在する顧客の場合は削除を拒否し、エラーメッセージを表示する

**なぜそうなっているのか:**
- 受注履歴を顧客詳細から見られるのは、「この顧客とどんな取引をしてきたか」を一画面で把握するため
- 受注が存在する顧客を削除できない（`CustomerHasOrdersException`）のは、受注データとの整合性を守るため。顧客を消すと受注の「誰の注文か」が失われてしまう

---

### 3-3. 顧客登録 `/customers/create` / 顧客編集 `/customers/{id}/edit`

**アクセス権限:** sales / admin（登録・編集とも）

#### 画面項目・バリデーション

| 項目名 | 必須 | バリデーションルール | なぜそのルールなのか |
|--------|------|---------------------|-------------------|
| 会社名 | ○ | 最大255文字 | DBカラムの上限に合わせている |
| 担当者名 | - | 最大255文字 | 任意項目。窓口担当者がいない場合もある |
| 住所 | - | 最大500文字 | 住所は長くなりがちなため余裕を持たせている |
| 電話番号 | - | 最大20文字（文字列型） | ハイフンあり/なし両方を受け付けるため数値ではなく文字列 |
| メールアドレス | - | メール形式・最大255文字 | フォーマット検証で入力ミスを防ぐ |
| 与信枠 | ○ | 整数・0以上 | 負の与信枠は業務上意味をなさないため |

#### 処理内容

1. **登録:** バリデーション通過後、`customers` テーブルに INSERT。完了後は顧客詳細画面へリダイレクトし「顧客を登録しました」と表示
2. **編集:** バリデーション通過後、対象レコードを UPDATE。完了後は顧客詳細画面へリダイレクト
3. **バリデーションエラー:** エラー時はフォームに入力値を保持したまま再表示（`old()` 関数で実現）

**なぜそうなっているのか:**
- 登録後に詳細画面へリダイレクトするのは、「今登録した内容を確認できる」UXのため（登録後に一覧へ戻るとどれが新しいか分かりにくい）
- ソフトデリート（`SoftDeletes`）を使っているのは、誰かが誤って削除した場合にDBレベルで復元できるようにするため（`deleted_at` カラムに削除日時が入り、通常クエリからは除外される）

## 4. 製品マスタ管理

### 4-1. 製品一覧 `/products`

**アクセス権限:** sales / accounting / warehouse / admin（全ロール）

#### 画面項目

| 項目名 | 説明 |
|--------|------|
| 検索ボックス | 品番・製品名で部分一致検索 |
| 品番 | 製品を一意に識別するコード（例: ABC-001） |
| 製品名 | 製品の名称 |
| 単価 | 標準販売単価（円）。カンマ区切り表示 |
| 単位 | 数量の単位（例: 個、kg、本） |
| 在庫数 | 実際に倉庫にある在庫数（`stock_quantity`） |
| 引当中 | 受注確定済みで出荷待ちの数量（`reserved_quantity`） |
| 利用可能 | `在庫数 - 引当中` = 実際に新規受注に使える数量 |
| 在庫状況バッジ | 在庫切れ（赤バッジ）/ 在庫不足（薄赤バッジ）/ 正常（表示なし） |
| 編集リンク | admin のみ表示 |
| 在庫調整リンク | warehouse / admin のみ表示 |

#### 処理内容

1. **在庫アラート表示:** `stock_quantity < alert_threshold` の場合、行の背景を赤く着色し「在庫不足」バッジを表示。`stock_quantity === 0` の場合は「在庫切れ」バッジを表示
2. **利用可能在庫の計算:** `stock_quantity - reserved_quantity` をビュー側でリアルタイム計算して表示

**なぜそうなっているのか:**
- 「在庫数」と「引当中」を別々に表示しているのは、引当済みの分は実際には出荷予定の在庫なので、新しい受注に使えないことを明確にするため
- 在庫アラートの閾値（`alert_threshold`）を製品ごとに設定できるのは、製品によって発注リードタイムや最低在庫量が異なるため

---

### 4-2. 製品登録 `/products/create` / 製品編集 `/products/{id}/edit`

**アクセス権限:** admin のみ

#### 画面項目・バリデーション

| 項目名 | 必須 | バリデーションルール | なぜそのルールなのか |
|--------|------|---------------------|-------------------|
| 品番 | ○ | 最大50文字・**システム全体で一意** | 同じ品番の製品が2つ存在するとどちらの製品か特定できなくなるため |
| 製品名 | ○ | 最大255文字 | DBカラム上限 |
| 単価 | ○ | 整数・0以上 | 負の単価は業務上ありえないため |
| 単位 | ○ | 最大20文字 | 「個」「kg」「本」など短い単位を想定 |
| 在庫数 | ○ | 整数・0以上 | 初期在庫数。マイナス在庫は管理上不正のため |
| 在庫アラート閾値 | ○ | 整数・0以上 | この数を下回ると一覧でアラート表示される |

**なぜ製品登録が admin のみなのか:**
製品マスタは全ロールが参照するマスタデータのため、誤った製品情報が登録されると見積・受注すべてに影響が出る。そのため管理者だけが変更できるよう制限している。

---

### 4-3. 在庫調整 `/products/{id}/adjust-stock`

**アクセス権限:** warehouse / admin

#### 画面項目

| 項目名 | 説明 |
|--------|------|
| 製品情報表示 | 品番・製品名・実在庫数・引当中在庫数・利用可能在庫数（参照用） |
| 増減数 | プラスで増加、マイナスで減少。**0は入力不可** |
| 調整理由（メモ） | 任意。なぜ調整するかの記録（例: 「棚卸しにより実数と合わせた」） |

#### 処理内容

1. **在庫更新:** `stock_quantity` を増減数分だけ加算（マイナス値なら減算）
2. **在庫変動履歴の記録:** `stock_movements` テーブルに `reason = manual_adjustment（5）` として操作ログを自動記録
3. **整合性チェック:** 調整後に在庫が「引当中在庫数」を下回る場合は `StockAdjustmentViolatesIntegrityException` をスローしてエラーを返す
4. **二重送信防止:** 「調整を実行する」ボタンは押下後にdisabledになり「処理中...」と表示される

**なぜそうなっているのか:**
- 在庫変動をすべてログに残すのは、「いつ・誰が・何個増減したか」を追跡できるようにするため（棚卸しの監査対応、ヒューマンエラーの調査）
- 0を入力不可にしているのは、0の調整は意味がなくログだけが増える不要操作のため
- 引当中を下回る減算を禁止しているのは、「受注確定済みなのに在庫がない」矛盾状態を防ぐため

## 5. ユーザー管理

### 5-1. ユーザー一覧 `/admin/users`

**アクセス権限:** admin のみ

#### 画面項目

| 項目名 | 説明 |
|--------|------|
| 名前 | ユーザーの氏名 |
| メールアドレス | ログインに使用するメールアドレス |
| 役割 | ロール（システム管理者 / 営業担当者 / 在庫・出荷担当者 / 管理職・経理担当） |
| 状態 | 有効（緑バッジ）/ 無効（グレーバッジ） |
| 編集リンク | ユーザー編集画面へ遷移 |
| 無効化 / 有効化ボタン | アカウントを無効化・有効化する |
| パスワード再設定メール送信ボタン | 対象ユーザーにパスワードリセットメールを送る |

#### 処理内容

1. **一覧表示:** `id` 昇順で50件ページネーション
2. **有効化 / 無効化:** `is_active` フラグを反転。**自分自身は無効化できない**（システムロックアウト防止）
3. **パスワード再設定メール:** Laravelの組み込みパスワードリセット機能（`Password::sendResetLink`）を利用して対象ユーザーのメールアドレスにリセットリンクを送信

**なぜそうなっているのか:**
- ユーザーを「削除」でなく「無効化」で管理するのは、過去の操作ログや受注データにユーザー情報が紐づいているため。削除すると履歴の「誰が操作したか」が消えてしまう
- 自分自身を無効化できない制限は、管理者が自分のアカウントを誤って無効化するとシステムに誰もログインできなくなるリスクを防ぐため
- パスワード再設定をシステム管理者が任意のタイミングで送れるのは、社員がパスワードを忘れた際のサポートオペレーション（ヘルプデスク対応）のため

---

### 5-2. ユーザー登録 `/admin/users/create` / ユーザー編集 `/admin/users/{id}/edit`

**アクセス権限:** admin のみ

#### 画面項目・バリデーション

| 項目名 | 必須 | バリデーションルール | なぜそのルールなのか |
|--------|------|---------------------|-------------------|
| 名前 | ○ | 最大255文字 | 表示名として使用するため |
| メールアドレス | ○ | メール形式・小文字・**システム全体で一意** | ログインIDとして使用するため重複不可 |
| パスワード | ○（登録時のみ） | 確認フィールドと一致・Laravelデフォルト強度要件 | 入力ミス防止と最低限のセキュリティ確保 |
| 役割 | ○ | 有効なロール値のいずれか | 不正なロール値でアカウントが作られないように |

#### ロール説明の動的表示

役割を選択すると、その役割の説明文がリアルタイムで表示される（JavaScriptで実装）。

**なぜそうなっているのか:**
- 初期パスワードは管理者が設定するが、登録後すぐにパスワード再設定メールを送る運用を想定しているため、強度要件は必要最低限にとどめている
- 役割の説明をUI上で見せているのは、管理者がロールの違いを把握した上で正しく割り当てられるようにするため

## 6. 見積管理

### 6-1. 見積一覧 `/quotations`

**アクセス権限:** sales / admin

#### 画面項目

| 項目名 | 説明 |
|--------|------|
| 見積番号 | `QUO-{年度}-{連番4桁}` 形式（例: QUO-2026-0001） |
| 顧客名 | 見積の宛先顧客 |
| ステータス | DRAFT / CONVERTED / CANCELLED / EXPIRED |
| 有効期限 | 設定した場合は `YYYY-MM-DD` 形式。なければ「—」 |
| 詳細リンク | 見積詳細画面へ遷移 |
| 見積新規作成ボタン | 見積作成画面へ遷移 |

#### 処理内容

1. **一覧取得:** 50件ページネーション（デフォルト）
2. **ステータス表示:** 値はEnumの名前（DRAFT, CONVERTED等）をそのまま表示

---

### 6-2. 見積新規作成 `/quotations/create`

**アクセス権限:** sales / admin

#### 画面項目

| 項目名 | 必須 | バリデーションルール | 説明 |
|--------|------|---------------------|------|
| 顧客 | ○ | `customers` テーブルに存在すること | プルダウンから選択 |
| 明細（製品） | ○（1行以上） | `products` テーブルに存在すること | プルダウンから選択 |
| 明細（数量） | ○ | 整数・1以上 | マイナスや0の数量は受注として意味をなさないため |
| 明細（単価） | ○ | 整数・0以上 | 無料提供（0円）は許容するため0以上 |
| 有効期限 | - | 日付形式 | 任意。設定した日付を過ぎると受注確定不可になる |
| 備考 | - | 最大1000文字 | 顧客への伝達事項など |

#### 処理内容

1. **明細行の動的追加・削除:** 「明細行を追加」ボタンで行を増やせる。「削除」ボタンで行を削除できる（JavaScriptで実装）
2. **金額のリアルタイム計算:** 数量・単価を入力するたびに内部APIエンドポイント（`POST /api/internal/quotations/calculate`）を呼び出し、各明細の金額と合計金額をリアルタイム更新する
3. **製品選択時の単価自動入力:** 製品プルダウンを選択すると、その製品のマスタ単価が単価フィールドに自動入力される（JavaScriptで実装）
4. **見積番号の自動採番:** 保存時に `QUO-{西暦年}-{連番4桁}` 形式で見積番号を発行する。連番は `document_sequences` テーブルで管理されており、採番処理はトランザクション内で実行される
5. **保存後:** 作成した見積の詳細画面へリダイレクト

**なぜそうなっているのか:**
- 単価を「製品マスタの単価を初期値として手動変更可能」にしているのは、顧客ごとの値引き交渉や特別価格に対応するため（製品マスタの価格を直接変えなくて済む）
- 金額計算をサーバー側APIで行っているのは、計算ロジックをJavaScriptだけに任せると改ざんリスクがあるため
- 見積番号採番をDBトランザクション内で行っているのは、同時に複数の見積が作られた際に番号が重複しないようにするため

---

### 6-3. 見積詳細 `/quotations/{id}`

**アクセス権限:** sales / admin

#### 画面項目

| 項目名 | 説明 |
|--------|------|
| 見積番号 | ヘッダーとして表示 |
| 顧客 | 宛先顧客名 |
| ステータス | 現在のステータス |
| 有効期限 | 設定されていれば表示 |
| 備考 | 設定されていれば表示 |
| 明細テーブル | 製品名・数量・単価・金額（数量×単価） |
| 合計金額 | 全明細の金額合計 |
| PDFプレビュー・ダウンロードボタン | 見積書PDFを生成してダウンロード |
| **受注確定ボタン** | 見積を受注に転換する |

#### 処理内容（受注確定）

受注確定ボタン押下後、以下の処理が**DBトランザクション内でアトミックに**実行される:

```
1. 有効期限チェック
   └ 期限切れの場合 → ステータスを EXPIRED に更新してエラー表示（受注確定しない）

2. 在庫チェック（悲観的ロック）
   └ 各明細の製品を行ロック（SELECT FOR UPDATE）
   └ 利用可能在庫（stock_quantity - reserved_quantity）< 明細数量 の製品があれば InsufficientStockException をスロー
   └ 不足製品が1つでもあれば全体をロールバック → エラーメッセージ表示

3. 在庫引当
   └ 各明細の製品の reserved_quantity を明細数量分だけ加算
   └ stock_movements に reason=RESERVATION のログを記録

4. 受注作成
   └ sales_orders テーブルに ORD-{年度}-{連番4桁} 形式の受注番号で作成
   └ sales_order_items テーブルに明細をコピー

5. 見積ステータス更新
   └ quotations.status を CONVERTED（受注転換済み）に更新
```

**なぜそうなっているのか:**
- 在庫チェックに**悲観的ロック（SELECT FOR UPDATE）**を使っているのは、同時に2人が同じ在庫を確定しようとした場合の二重引当を防ぐため（楽観的ロックだと競合した場合に在庫がマイナスになりうる）
- 有効期限切れの場合に受注確定を拒否するのは、古い見積書の価格・条件で誤って受注してしまうリスクを防ぐため
- 受注確定後に見積書PDFをダウンロードできるのは、顧客との合意内容を書面として保管するため

---

### 6-4. 見積PDF出力 `/quotations/{id}/pdf`

**アクセス権限:** sales / admin

見積情報をもとにPDFを生成し、ブラウザへダウンロードレスポンスとして返す。ファイル名は `{見積番号}.pdf`（例: `QUO-2026-0001.pdf`）。

## 7. 見積ステータス遷移図

```
         ┌─────────────────────────────────────┐
         │           見積作成時                  │
         ▼                                       │
  ┌─────────────┐                               │
  │   DRAFT      │  ← 初期ステータス              │
  │  （見積中）   │                               │
  └─────┬───────┘                               │
        │                                        │
        ├─── 受注確定操作 ──────► CONVERTED        │
        │                        （受注転換済み）    │
        │                                        │
        ├─── 有効期限切れで                        │
        │    受注確定を試みる ──► EXPIRED           │
        │                        （期限切れ）       │
        │                                        │
        └─── ※ CANCELLED は現在                  │
              画面から操作不可（将来実装予定）         │
```

| ステータス | 値 | 意味 | 受注確定可否 |
|-----------|-----|------|------------|
| DRAFT | 1 | 見積中 | 可（在庫があれば） |
| CONVERTED | 2 | 受注転換済み | 不可（すでに受注化済み） |
| CANCELLED | 3 | キャンセル | 不可 |
| EXPIRED | 4 | 期限切れ | 不可 |

## 8. 在庫の考え方

本システムでは在庫を3つの数値で管理している。

```
stock_quantity（実在庫数）
    │
    ├─ reserved_quantity（引当中在庫数）
    │      ← 受注確定時に加算。出荷完了時に減算（出荷機能は別タスク）
    │
    └─ 利用可能在庫 = stock_quantity - reserved_quantity
           ← 新規受注の際にこの数量で在庫チェックを行う
```

**具体例:**
- 実在庫: 100個
- 引当中: 30個（受注確定済みだが未出荷）
- 利用可能: 70個 ← この製品で新規に受注できる最大数量

**なぜこの3段階管理が必要か:**
単純に `stock_quantity` だけで管理すると、「受注確定済みの在庫」と「新しく受注できる在庫」の区別がつかない。例えば在庫100個で100個の受注を確定した後に、さらに50個の受注が入った場合、在庫チェックは「100 >= 50 → OK」と判断してしまう（実際は引当済みで0個しか使えない）。引当数を別管理することでこの問題を防いでいる。

## 9. 用語集

| 用語 | 説明 |
|------|------|
| **マスタデータ** | 顧客・製品など、システムの基盤となる基本情報。取引のたびに変わらない |
| **与信枠** | 顧客に対して許容する取引上限金額。この金額を超えた受注は与信管理上の警告対象 |
| **見積書** | 顧客に「この製品をこの価格で提供できます」と提示する書類 |
| **受注** | 顧客が見積を承認し、正式に注文した状態 |
| **在庫引当** | 受注確定時に「この在庫は出荷予定」として確保する処理。他の受注では使えなくなる |
| **ソフトデリート** | DBのレコードを実際には削除せず `deleted_at` に削除日時を記録する方式。復元可能 |
| **ページネーション** | 大量のデータを複数ページに分けて表示する仕組み |
| **バリデーション** | フォーム送信されたデータが正しい形式・値かをサーバー側で検証する処理 |
| **トランザクション** | 複数のDB操作を「すべて成功」か「すべて失敗（ロールバック）」のどちらかにまとめる仕組み |
| **悲観的ロック** | 「他の処理がデータを変更するかもしれない」と仮定してロックをかける方式。在庫引当で使用 |
| **ロール** | ユーザーの役割・権限区分（admin / sales / warehouse / accounting） |
| **CRUD** | Create（作成）/ Read（参照）/ Update（更新）/ Delete（削除）の略 |
