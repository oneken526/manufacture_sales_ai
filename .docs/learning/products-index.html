前提知識：在庫の3段階管理

本システムでは在庫を以下の3つの数値で管理しています。

stock_quantity（実在庫数） │ ├─ reserved_quantity（引当中）← 受注確定時に加算。出荷完了で減算 │ └─ 利用可能在庫 = stock_quantity - reserved_quantity ← 新規受注の際にこの数量でチェックを行う

例：実在庫100個、引当中30個 → 利用可能70個（この製品で受注確定できる最大数量）

処理内容
  • 検索フィルタの適用
    URLパラメータ ?q=キーワード が存在する場合、品番・製品名に対して LIKE '%キーワード%' で絞り込む。
  • 在庫アラートの判定
    各製品について stock_quantity と alert_threshold を比較してバッジと背景色を決定する。この判定はビュー側（Blade）で行う。
  • 利用可能在庫の計算
    stock_quantity - reserved_quantity をビュー側でリアルタイム計算して表示する。DBに保存はしない。
  • ページネーション
    1ページあたり50件表示。検索キーワードはページ遷移時に引き継がれる。
💡 なぜそうなっているのか
  • 「在庫数」と「引当中」を別々に表示する理由: 引当済みの在庫は出荷予定分なので新しい受注に使えない。「実際に使える在庫」を正確に伝えるためこの3つを並べて表示している。
  • 在庫アラートの閾値を製品ごとに設定できる理由: 製品によって発注リードタイムや最低在庫量が異なるため。「この製品は10個割ったらアラート、あちらは50個割ったらアラート」という運用が必要。
  • 製品一覧が全ロールに開放されている理由: 製品情報（在庫数・単価など）は営業・倉庫・経理すべての部門が参照するため。参照は誰でも見られるが、変更は管理者だけに絞っている。
