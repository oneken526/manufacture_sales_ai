ユーザー登録・編集
システムユーザーの新規登録と情報更新を行う画面（管理者専用）
URL（登録） /admin/users/create
URL（編集） /admin/users/{id}/edit
アクセス権限 admin のみ
画面項目とバリデーション
項目名 登録 編集 バリデーションルール なぜそのルールなのか
名前 ● ● 文字列・最大255文字 画面上の表示名として使用するため必須
メールアドレス ● ● メール形式・小文字・システム全体で一意 ログインIDとして使用するため重複不可。小文字に統一することで大文字小文字の違いによるログイン失敗を防ぐ
パスワード ● —（編集不可） 確認フィールドと一致・Laravelデフォルト強度要件 入力ミス防止のため確認入力を必須とする。編集では変更しない（パスワード変更はリセットメールで行う）
役割 ● ● 有効なロール値（admin / sales / warehouse / accounting）のいずれか 不正なロール値でアカウントが作られないようにサーバーサイドで検証する
役割プルダウンの動的説明文

役割プルダウンを選択すると、JavaScriptによってその役割の説明文がリアルタイムで表示されます。管理者がロールの意味を理解した上で正しく割り当てられるようにするための機能です。

ロール値 表示される説明文
admin システム管理者：すべての機能（マスタ管理、ユーザー管理、運用機能）を利用できます。
sales 営業担当者：顧客・受注・見積・請求書の作成や編集ができます。
warehouse 在庫・出荷担当者：在庫管理・出荷処理ができます。請求書操作はできません。
accounting 管理職・経理担当：請求書発行や経営状況の確認ができます。
処理内容
  • バリデーション（サーバーサイド）
    StoreUserRequest / UpdateUserRequest のルールに基づきチェック。エラー時は入力値を保持したまま再表示する。
  • 登録処理（新規登録の場合）
    パスワードをbcryptでハッシュ化（Hash::make()）してから users テーブルに INSERT。is_active = true で有効状態として登録。成功後はユーザー一覧画面へリダイレクト。
  • 更新処理（編集の場合）
    名前・メールアドレス・役割を UPDATE する。パスワードは編集画面から変更できない（パスワード変更はユーザー一覧の「パスワード再設定メール送信」から行う）。成功後は編集画面へリダイレクト。
💡 なぜそうなっているのか
  • パスワードを編集画面で変更できない理由: 管理者が他人のパスワードを直接知ることはセキュリティリスクになる。パスワードリセットメール経由にすることで、本人だけが新しいパスワードを設定する仕組みを維持している。
  • 役割説明をUIで見せる理由: 管理者がロールの違いを正確に理解した上でアカウントを作成できるようにするため。間違ったロールでアカウントを作ると、過不足のある権限でシステムを使われてしまう。
  • メールアドレスを小文字に統一する理由: ABC@example.com と abc@example.com は同一メールアドレスだが、大文字小文字の違いでDBが別レコードと判断するのを防ぐため。
