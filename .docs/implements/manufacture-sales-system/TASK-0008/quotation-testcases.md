### TC6: 在庫不足時にInsufficientStockExceptionがスローされ確定処理が中断される（EDGE-001）
- **テスト名**: 在庫不足時に例外がスローされロールバックされる
  - **エラーケースの概要**: 見積明細のいずれかの製品で、利用可能在庫（`stock_quantity - reserved_quantity`）が明細数量を下回っている状態で受注確定を試みる

  - **実際の発生シナリオ**: 複数の営業担当者が同時期に同一製品を見積もり、後から確定しようとした際に在庫が他の受注で引き当て済みになっているケース

  - **エラーメッセージの内容**: 「在庫が不足しています（製品: ○○、要求数量: 8、利用可能数量: 5）」のように具体的な製品名・数量を含む
  - **システムの安全性**: トランザクション内で例外をスローすることで、一部だけ更新された中途半端な状態を防ぎ、DBの整合性を保つ
- **テストの目的**: 在庫不足時のエラーハンドリングが要件通り機能し、データ不整合を防止できることを確認する
  - **品質保証の観点**: EDGE-001はコア機能の安全性に直結するエッジケースであり、確実にテストでカバーする必要がある

## 3. 境界値テストケース（最小値、最大値、null等）

### TC10: 利用可能在庫と明細数量が完全に一致する場合は受注確定が成功する
- **テスト名**: 利用可能在庫＝要求数量という境界条件で受注確定が成功する
  - **境界値の意味**: 「在庫不足」と判定される境界線（利用可能在庫 < 要求数量）のすぐ外側（等しい場合）であり、誤って不足と判定されないことを保証する重要な境界
  - **境界値での動作保証**: 「不足」の判定が`<`（より小さい）であり`<=`ではないことを保証し、ぎりぎりの在庫でも正しく受注できることを確認する
- **入力値**: 製品の`stock_quantity=10, reserved_quantity=0`（利用可能在庫10）に対し、明細`quantity=10`を指定
  - **境界値選択の根拠**: 利用可能在庫と要求数量が等しい(10=10)ちょうど境界の値を選定し、不足ではなく成功すべきことを示す
  - **実際の使用場面**: 在庫がぎりぎりの製品を全量受注するケース（製造業では在庫切れ間際の受注が頻繁に発生する）
- **期待される結果**: 例外はスローされず、`reserved_quantity`が10加算（0→10）され、受注確定処理が正常に完了する
  - **境界での正確性**: `利用可能在庫 < 要求数量`の判定式が正しく実装されていれば、等しい場合は「不足ではない」と判定される
  - **一貫した動作**: 境界値の内側（在庫9で要求10＝不足）と外側（在庫10で要求10＝成功）で動作が一貫して切り替わることを保証する
- **テストの目的**: 在庫充足判定の境界条件（等号の扱い）が要件通り実装されていることを確認する
  - **堅牢性の確認**: オフバイワンエラー（off-by-one）が発生していないことを保証する
- 🔵 信頼性レベル: EDGE-001の判定式（利用可能在庫 = stock_quantity - reserved_quantity が要求数量を下回る場合に警告）から導出される境界値であり、要件定義書の記述から直接特定できる

### TC11: 利用可能在庫が要求数量より1だけ少ない場合は在庫不足と判定される
- **テスト名**: 利用可能在庫が要求数量を1つ下回る境界条件で在庫不足エラーとなる
  - **境界値の意味**: TC10の対となる境界値であり、「成功」と「失敗」の境界線のすぐ内側（不足側）を検証する
  - **境界値での動作保証**: 1つでも不足していれば確実に例外がスローされ、在庫超過の受注が成立しないことを保証する
- **入力値**: 製品の`stock_quantity=9, reserved_quantity=0`（利用可能在庫9）に対し、明細`quantity=10`を指定
  - **境界値選択の根拠**: 利用可能在庫(9)が要求数量(10)よりちょうど1少ない、最小の不足幅を選定
  - **実際の使用場面**: 在庫がわずかに足りない製品を受注しようとした際、確実にエラーとして検出されるべき典型的なケース
- **期待される結果**: `InsufficientStockException`がスローされ、例外情報に要求数量10・利用可能数量9が含まれる
  - **境界での正確性**: わずか1の差でも正しく不足として検出される
  - **一貫した動作**: TC10（境界の外側=成功）とTC11（境界の内側=失敗）が対になって動作の一貫性を証明する
- **テストの目的**: 在庫充足判定が最小の不足幅でも正しく機能することを確認する
  - **堅牢性の確認**: 判定ロジックに誤差や丸め誤差が存在しないことを保証する
- 🔵 信頼性レベル: EDGE-001の判定式から導出される境界値（TC10と対）であり、要件定義書の記述から直接特定できる

### TC12: document_sequencesの採番が同時実行でも重複しない（排他制御）
- **テスト名**: 複数リクエストが同時に見積番号採番を行っても重複・抜けが発生しない
  - **境界値の意味**: 通常は単一プロセスでの逐次実行を想定するが、複数の営業担当者が同時に見積を保存する「並行アクセス」という特殊な実行条件の境界を検証する
  - **境界値での動作保証**: 同時実行という特殊条件下でも、`lockForUpdate()`による排他制御により採番が直列化され、一貫した結果が得られることを保証する
- **入力値**: 同一年度（例: 2026年度）に対して、複数（例: 5件）の見積保存処理を並行して実行する
  - **境界値選択の根拠**: 同時実行数を複数に設定することで、ロックが機能しない場合に重複が発生しやすい状況を作り出す
  - **実際の使用場面**: 月初・年度末など繁忙期に複数の営業担当者が同時に見積を作成・保存するシーン
- **期待される結果**: 生成される見積番号（`QUO-2026-0001`〜`QUO-2026-0005`）が重複せず、`document_sequences.last_number`が呼び出し回数分（5）だけ正しくインクリメントされる
  - **境界での正確性**: `lockForUpdate()`がトランザクション内で正しく機能し、行ロックが直列化を保証する
  - **一貫した動作**: 同時実行数に関わらず、最終的な`last_number`と発行された番号の集合が一致する
- **テストの目的**: 悲観的ロックによる排他制御が並行処理下でも機能することを確認する
  - **堅牢性の確認**: 競合状態（race condition）によるデータ不整合が発生しないことを保証する
- 🔵 信頼性レベル: TASK-0008.md単体テスト要件テストケース3・設計ヒアリングQ5・database-schema.sql（UNIQUE制約・悲観的ロック方針）より直接抽出

### TC13: 明細が0件の見積詳細画面では受注確定ボタンが非活性化される（EDGE-011）
- **テスト名**: 明細0件の見積詳細画面で受注確定ボタンがdisabled状態で表示される
  - **境界値の意味**: 明細件数の最小値（0件）という境界条件における画面表示の振る舞いを検証する
  - **境界値での動作保証**: 明細が1件以上ある場合とない場合とでボタンの活性状態が正しく切り替わることを保証する
- **入力値**: `quotation_items`が0件の見積（明細未登録の状態。例えば過去データやテスト用に直接作成したケース）の詳細画面表示
  - **境界値選択の根拠**: 「ボタンが活性化されるか否か」を分岐させる最小の境界（0件 vs 1件以上）を選定
  - **実際の使用場面**: データ移行や手動操作によって明細のない見積データが存在してしまった場合の画面表示
- **期待される結果**: 詳細画面のHTMLに「受注確定」ボタンの`disabled`属性が含まれる（`assertSee('disabled', false)`相当のHTML検証）。非活性の理由（「明細が登録されていないため受注確定できません」）がツールチップまたは補足テキストとして表示される
  - **境界での正確性**: 明細件数のカウントが正確に行われ、0件の場合のみボタンが非活性化される
  - **一貫した動作**: 明細1件以上のケース（TC3等）ではボタンが活性化されることと対比して一貫性を確認する
- **テストの目的**: EDGE-011の要件（明細0件時のボタン非活性化）がUIレベルで実装されていることを確認する
  - **堅牢性の確認**: 境界値（0件）でのUI制御が確実に機能し、不正な受注確定操作を未然に防ぐ
- 🟡 信頼性レベル: TASK-0008.md完了条件「明細が0件の場合に受注確定ボタンが非活性化されること（EDGE-011）」・要件定義書でも🟡（UI/UXから推測）と明記
