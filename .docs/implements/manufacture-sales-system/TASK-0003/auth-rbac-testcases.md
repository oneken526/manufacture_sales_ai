## 3. 境界値テストケース（最小値、最大値、null等）

### TC-B-01: roleミドルウェアに許可ロールを1つだけ指定した場合の境界動作
- **テスト名**: `role:admin`（許可ロールが単一）の場合、ADMINは通過しSALES等は403となる
  - **境界値の意味**: ミドルウェアの可変長引数`...$roles`が最小要素数（1個）の場合に正しく動作するかという実装の境界
  - **境界値での動作保証**: 複数ロール指定時（TC-N-02, TC-A-01）と単一ロール指定時とで判定ロジックが一貫していることを保証する
- **入力値**: `role:admin`を付与したテスト用ルートに対し、`role=ADMIN`のユーザーと`role=SALES`のユーザーでそれぞれアクセス
  - **境界値選択の根拠**: `string ...$roles`は0個以上を受け取れる可変長引数だが、実運用では最低1個のロール指定が必須となるため「1個」が実質的な最小境界となる
  - **実際の使用場面**: `routes/web.php`で`->middleware(['auth', 'role:admin'])`のように単一ロール限定のルートを定義する場面（例: ユーザー管理画面はADMIN限定）
- **期待される結果**: ADMINユーザーは`assertOk()`（通過）、SALESユーザーは`assertForbidden()`（403）
  - **境界での正確性**: `in_array($request->user()->role->value, $roles)`等の判定が、要素数1の配列に対しても正しく真偽を返す
  - **一貫した動作**: 複数ロール指定（TC-N-02/TC-A-01）と単一ロール指定とで拒否・許可の判定基準が変わらない
- **テストの目的**: 可変長引数の要素数に依存しない安定したロール判定の確認
  - **堅牢性の確認**: ルート定義側の指定方法（単一/複数）によらずミドルウェアが正しく機能すること
- 🟡 信頼性レベル: TASK-0003.mdの例は複数ロール指定（`role:admin,accounting`）が中心のため、単一ロール指定時の挙動は可変長引数の仕様から導かれる妥当な推測

### TC-B-02: `users.role`の境界値（Enum全4値）でのキャスト・判定の一貫性
- **テスト名**: `role`カラムの値1〜4（ADMIN/SALES/WAREHOUSE/ACCOUNTING）すべてが`UserRole` Enumに正しくキャストされる
  - **境界値の意味**: `database-schema.sql`のCHECK制約`role BETWEEN 1 AND 4`が示す値域の最小値（1）・最大値（4）を含む全区分値が境界
  - **境界値での動作保証**: DBに保存された数値コードとアプリケーション層のEnum値が全件で一致し、ロール判定・ミドルウェア・Gateの全機構に正しく伝播することを保証する
- **入力値**: `role`カラムに`1, 2, 3, 4`をそれぞれ持つ4件の`User`をDBに保存し、Eloquentで取得
  - **境界値選択の根拠**: CHECK制約で許容される値域の全パターン（最小値1=ADMIN、最大値4=ACCOUNTING、中間値2=SALES, 3=WAREHOUSE）を網羅する
  - **実際の使用場面**: シーダーや管理画面経由でユーザーが作成され、DBに整数値として保存・取得される一連の流れ
- **期待される結果**: 取得した各`User`の`role`プロパティが、それぞれ`UserRole::ADMIN`, `UserRole::SALES`, `UserRole::WAREHOUSE`, `UserRole::ACCOUNTING`のEnumインスタンスとなり、`->value`が`1,2,3,4`と一致する
  - **境界での正確性**: 最小値・最大値を含むすべての区分値で取りこぼしなくキャストされる
  - **一貫した動作**: 4値のいずれであってもミドルウェア・Gate・`hasRole`が同一ロジックで正しく判定する
- **テストの目的**: `UserRole::class`への属性キャスト（TASK-0002で設定済み）が認証・権限基盤全体の前提として正しく機能し続けていることの確認
  - **堅牢性の確認**: 既存のEnumキャスト実装（`app/Models/User.php`）に対する回帰確認を兼ね、本タスクの権限制御がこの前提の上に安全に積み上がっていることを保証する
- 🔵 信頼性レベル: `database-schema.sql`のCHECK制約定義および`app/Enums/UserRole.php`の実装内容に直接基づく
