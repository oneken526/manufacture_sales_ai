# TASK-0004: PDFサービス基盤実装 - テストケース

## 1. 正常系テストケース

### TC1: BladeビューからPDFバイナリを生成できる
- **テスト名**: PdfServiceがBladeビューからPDFバイナリを生成できる
  - **何をテストするか**: `generateFromView()` がBladeビューとデータからPDFバイナリを生成して返すこと
  - **期待される動作**: テンプレートが正常にレンダリングされ、mPDFによりPDFバイナリへ変換される
- **入力値**: `pdf.test-template`（テスト用簡易テンプレート）, `['title' => 'テスト帳票', 'issuedAt' => '2026-06-07']`
  - **入力データの意味**: 後続帳票（見積/納品/請求）と同様にBladeテンプレート＋連想配列で渡す代表的なパターン
- **期待される結果**: 戻り値の文字列が `%PDF-` で始まり、空でないバイト列であること
  - **期待結果の理由**: PDFファイルのマジックナンバー（シグネチャ）が `%PDF-` であるため、これが含まれることでPDFとして有効なバイナリであると判定できる
- **テストの目的**: Blade→PDF変換のコア機能が動作することを保証する
  - **確認ポイント**: 戻り値が文字列型であること、PDFシグネチャを含むこと、サイズが0でないこと
- 🔵 信頼性レベル: タスクファイル単体テストケース1・要件定義2章に基づく

### TC2: 日本語が文字化けせずPDFに表示される
- **テスト名**: 日本語（漢字・ひらがな・カタカナ）を含むテンプレートが正しくPDF化される
  - **何をテストするか**: `config/mpdf.php` の日本語フォント設定（meiryo）が適用され、日本語テキストが正しくPDFに埋め込まれること
  - **期待される動作**: 生成されたPDFのテキストを抽出すると、入力した日本語文字列と一致する
- **入力値**: `['message' => '見積書 株式会社サンプル製作所 御中 ㈱テスト 単価：￥1,000']`（漢字・ひらがな・カタカナ・記号混在）
  - **入力データの意味**: 帳票で実際に使われる代表的な日本語表現（会社名・敬称・金額表記）を網羅する
- **期待される結果**: 生成PDFバイナリから抽出したテキストに入力した日本語文字列が含まれること（文字化け＝代替文字や空文字になっていないこと）
  - **期待結果の理由**: 文字化けするとテキスト抽出結果が異なるバイト列・代替文字になるため、元の文字列と一致するかで判定できる
- **テストの目的**: 日本語帳票対応（design-interview.md Q2）の動作保証
  - **確認ポイント**: PDFバイナリのサイズが妥当であること、抽出テキストに日本語が正しく含まれること
- 🔵 信頼性レベル: タスクファイル単体テストケース2・design-interview.md Q2に基づく

### TC3: ファイル名・保存パスが命名規則どおりに生成される（見積書）
- **テスト名**: 帳票種別「quotation」と識別子からパスが命名規則どおりに生成される
  - **何をテストするか**: パス生成ヘルパーが `storage/app/pdf/{帳票種別}/{年度}/{ファイル名}.pdf` 形式の文字列を返すこと
  - **期待される動作**: 帳票種別・識別子・年度から一貫したルールでパスが組み立てられる
- **入力値**: 帳票種別 `quotation`、識別子 `Q-2026-0001`、年度 `2026`
  - **入力データの意味**: 見積書の採番番号を代表する識別子。ファイル名規則 `quotation_{採番番号}.pdf` の検証に必要
- **期待される結果**: `pdf/quotation/2026/quotation_Q-2026-0001.pdf`（Storageディスク基準の相対パス）が返ること
  - **期待結果の理由**: タスク詳細「ファイル名の命名規則」（見積書`quotation_{採番番号}.pdf`）と「`storage/app/pdf/{帳票種別}/{年度}/`」のディレクトリ構成に一致するため
- **テストの目的**: 後続タスク（TASK-0008/0010/0012）が一貫した方法でパスを取得できることを保証する
  - **確認ポイント**: ディレクトリ階層・ファイル名の形式・拡張子が規則どおりであること
- 🟡 信頼性レベル: タスクファイル単体テストケース4（命名規則は推測を含む🟡指定）に基づく

### TC4: ファイル名・保存パスが命名規則どおりに生成される（納品書・請求書）
- **テスト名**: 帳票種別「shipment」「invoice」でも命名規則どおりにパスが生成される
  - **何をテストするか**: 帳票種別ごとに異なるファイル名プレフィックス（`shipment_`, `invoice_`）が正しく適用されること
  - **期待される動作**: 帳票種別を切り替えても共通のディレクトリ構成・命名ロジックで一貫したパスが得られる
- **入力値**: (`shipment`, `S-2026-0010`) → 期待 `pdf/shipment/2026/shipment_S-2026-0010.pdf`、(`invoice`, `I-2026-0099`) → 期待 `pdf/invoice/2026/invoice_I-2026-0099.pdf`
  - **入力データの意味**: 納品書の出荷ID・請求書の請求書番号を代表する識別子
- **期待される結果**: それぞれ規則どおりのパス文字列が返る
  - **期待結果の理由**: タスク詳細に明記された命名規則（`shipment_{出荷ID}.pdf`, `invoice_{請求書番号}.pdf`）に一致するため
- **テストの目的**: 帳票種別に依存しない汎用ヘルパーとして設計されていることを保証する
  - **確認ポイント**: `#[DataProvider]` でパラメータ化し、3帳票種別を網羅する
- 🟡 信頼性レベル: タスクファイル単体テストケース4に基づく（命名規則は推測🟡）

### TC5: 共通レイアウトを含むPDF生成→保存の統合フロー
- **テスト名**: PdfServiceを介してPDF生成からファイル保存・再取得までの一連の流れが動作する
  - **何をテストするか**: 共通レイアウト（ヘッダー・フッター・会社情報）を含むテンプレートからPDFを生成し、`storage/app/pdf/` 配下に保存、保存ファイルを再取得して内容を検証する一連の流れ
  - **期待される動作**: PDF生成・保存・再取得のいずれの工程でもエラーが発生せず、保存されたファイルがPDFとして開けること、ヘッダー・フッターに会社情報（`config/company.php` の会社名等）が含まれること
- **入力値**: `pdf.layouts.base` を継承した簡易テンプレート、`['companyName' => 'サンプル製作所株式会社', ...帳票データ]`
  - **入力データの意味**: 後続帳票テンプレートが共通レイアウトを利用する想定の代表的なシナリオ
- **期待される結果**: `Storage::disk('local')->exists($path)` が真となり、保存ファイルの内容が `%PDF-` で始まる。抽出テキストに会社名が含まれる
  - **期待結果の理由**: 統合テスト要件1の「ファイルとして開けること」「会社情報が含まれること」という期待結果に対応する
- **テストの目的**: `PdfService` 単体での生成→保存→再取得のEnd-to-Endフローを保証する
  - **確認ポイント**: 保存パスが命名規則と一致すること、ファイルが破損していないこと、共通レイアウトが適用されていること
- 🟡 信頼性レベル: タスクファイル統合テスト1（🟡：レイアウト内容は推測を含む）に基づく

## 2. 異常系テストケース

### TC6: 存在しないビュー名を指定した場合に例外がスローされる
- **テスト名**: 存在しないBladeビューを指定すると `PdfGenerationException` がスローされる
  - **エラーケースの概要**: 開発者の設定ミスやテンプレート削除等により、存在しないビュー名が渡されるケース
  - **エラー処理の重要性**: ビューレンダリング時の例外を生のまま伝播させるとユーザーに分かりにくいエラー画面が表示されてしまうため、独自例外でラップしてユーザー向けメッセージに変換する必要がある
- **入力値**: `'pdf.non-existent-template'`, `[]`
  - **不正な理由**: `resources/views/pdf/` 配下に存在しないビュー名であり、`view()->render()` が `InvalidArgumentException` 等を投げる
  - **実際の発生シナリオ**: テンプレートファイルの削除・リネーム、設定ミスにより発生しうる
- **期待される結果**: `PdfGenerationException` がスローされ、例外メッセージに「PDFの生成に失敗しました。しばらく経ってから再度お試しください」等、再試行を促す文言が含まれる。あわせて `Log::error()` でテンプレート名・例外メッセージが記録される
  - **エラーメッセージの内容**: ユーザーが次に取るべき行動（再試行）が明確に伝わる内容になっている
  - **システムの安全性**: 例外がアプリケーション全体を落とすことなく、Controller層で捕捉可能な独自例外型に変換されている
- **テストの目的**: EDGE-003（PDF生成失敗時のエラーメッセージ・再試行促進）の実装を保証する
  - **品質保証の観点**: 想定外のテンプレート参照ミスが発生してもユーザー影響を最小化し、運用調査に必要な情報がログに残ることを保証する
- 🟡 信頼性レベル: タスクファイル単体テストケース3・EDGE-003に基づく（実装詳細は推測🟡）

### TC7: テンプレートのレンダリングに失敗するデータを渡した場合に例外がスローされる
- **テスト名**: テンプレートが必要とする変数が渡されず、レンダリングエラーが発生すると `PdfGenerationException` がスローされる
  - **エラーケースの概要**: Controller側の実装ミスにより、テンプレートが参照する変数がデータ配列に含まれていないケース
  - **エラー処理の重要性**: Bladeのレンダリングエラー（未定義変数アクセス等）も他の生成エラーと同様にユーザー向けメッセージへ変換する必要がある
- **入力値**: `'pdf.test-template'`, `[]`（テンプレートが要求する `issuedAt` 等の変数を含まない空配列）
  - **不正な理由**: テンプレート内で `{{ $issuedAt }}` 等、未定義変数へアクセスし `ErrorException` が発生する
  - **実際の発生シナリオ**: 後続タスクでController実装時にデータ整形漏れが発生した場合に起こりうる
- **期待される結果**: `PdfGenerationException` がスローされ、再試行を促すメッセージを含む。`Log::error()` にテンプレート名・例外メッセージ・スタックトレースが記録される
  - **エラーメッセージの内容**: TC6と同様、ユーザーフレンドリーな再試行メッセージであること
  - **システムの安全性**: 内部エラーの詳細（スタックトレース等）はログにのみ記録され、ユーザーへ表示するメッセージには含まれない（個人情報・内部実装の非開示）
- **テストの目的**: テンプレートレンダリング起因のエラーも一貫した例外ハンドリングでラップされることを保証する
  - **品質保証の観点**: エラー要因（ビュー不在／レンダリング失敗／書込み失敗）によらず、ユーザー体験とログ出力が一貫することを保証する
- 🟡 信頼性レベル: タスクファイル単体テストケース3・実装詳細3に基づく（推測を含む🟡）

### TC8: PDF生成失敗が呼び出し元（Controller相当）まで伝播し画面表示・ログ記録される
- **テスト名**: モックでmPDFの例外を発生させた場合に `PdfGenerationException` が伝播し、ログに記録される
  - **エラーケースの概要**: mPDF内部処理（フォント読み込み・出力処理等）で予期しない例外が発生するケース
  - **エラー処理の重要性**: ライブラリ内部の例外も含めて統一的に独自例外へ変換し、Controller層・画面表示・ログ記録の一連の流れが機能することを確認する必要がある
- **入力値**: モック化した `Mpdf` インスタンスが `Output()` 呼び出し時に例外（例: `\Mpdf\MpdfException`）をスローするよう設定
  - **不正な理由**: 実際のフォント未配置・一時ディレクトリ書込み権限不足等により発生しうる例外を模擬する
  - **実際の発生シナリオ**: 本番環境でのフォントファイル未配置、ディスク容量不足によるファイル書き込み失敗等
- **期待される結果**: `PdfGenerationException` が伝播し、例外メッセージに再試行を促す文言が含まれる。`Log::error()` で詳細なエラー内容（テンプレート名・例外メッセージ・スタックトレース）が記録されている
  - **エラーメッセージの内容**: 統合テスト要件2の「ユーザーへ分かりやすいエラーメッセージ」に対応
  - **システムの安全性**: 例外発生後もアプリケーションが継続動作可能な状態であること（例外が握りつぶされず、かつクラッシュもしない）
- **テストの目的**: EDGE-003のエラーハンドリングフロー全体（生成失敗→例外伝播→ログ記録）が機能することを保証する
  - **品質保証の観点**: 障害発生時の運用調査に必要な情報が確実にログへ残り、ユーザーへの影響を最小化できることを保証する
- 🟡 信頼性レベル: タスクファイル統合テスト2（🟡：モックによる擬似的エラー）に基づく

## 3. 境界値テストケース

### TC9: データ配列が空の場合でもテンプレートに変数依存がなければ正常に生成できる
- **テスト名**: 空のデータ配列（`[]`）を渡しても、変数を参照しないテンプレートであればPDFが生成できる
  - **境界値の意味**: `generateFromView()` の `$data` 引数の最小値（空配列）を扱うケース。データなしでもクラッシュしないことを保証する
  - **境界値での動作保証**: データの有無によって処理分岐が生じないことを保証する
- **入力値**: `'pdf.static-template'`（変数を一切使わない静的HTMLのみのテンプレート）, `[]`
  - **境界値選択の根拠**: メソッドシグネチャ上 `array $data = []` がデフォルト値であり、最小・デフォルトの入力として妥当
  - **実際の使用場面**: 固定文言のみの帳票や注意書きページ等で発生しうる
- **期待される結果**: 例外を発生させず、`%PDF-` で始まるPDFバイナリが返却される
  - **境界での正確性**: データ有無に関わらず一貫してPDFバイナリが生成されること
  - **一貫した動作**: TC1（データありのケース）と同様の戻り値の型・形式であること
- **テストの目的**: メソッドのデフォルト引数・最小入力に対する堅牢性を確認する
  - **堅牢性の確認**: 空配列入力時にNullリファレンスや型エラーが発生しないことを確認する
- 🟡 信頼性レベル: メソッドシグネチャ（要件定義2章）からの妥当な推測🟡

### TC10: 年度をまたぐ識別子でもパス生成ロジックが一貫して動作する
- **テスト名**: 年初・年末の識別子に対してもパス生成の年度部分が正しく決定される
  - **境界値の意味**: 「年度」というディレクトリ階層の決定ロジックの境界（年度の切り替わり）を検証する
  - **境界値での動作保証**: 年度をまたぐ識別子・日時が混在しても、ディレクトリ構成が一貫したルールで決まることを保証する
- **入力値**: 識別子に年度情報を含むケース（例: `Q-2025-9999`）と、生成時点の年（例: 2026年）が異なるケース
  - **境界値選択の根拠**: 「年度」をどの情報（識別子内の年度／生成時点の年）から決定するかは設計判断であり、実装時に明確化・コメントで明記する必要があるため、その決定ロジックの一貫性を境界値として検証する
  - **実際の使用場面**: 年度をまたいで過去分の帳票PDFを再生成・再ダウンロードするケース
- **期待される結果**: 採用したルール（識別子由来か生成時点由来か）に従って一貫した年度ディレクトリが決定され、同じ識別子であれば常に同じパスが生成される（冪等性）
  - **境界での正確性**: 年度の決定方法が実装内で文書化され、テストがそのルールに対応していること
  - **一貫した動作**: 同一識別子に対して複数回呼び出しても同じパスが返ること（冪等性の確認）
- **テストの目的**: パス生成ロジックの年度決定ルールが明確かつ一貫していることを保証する
  - **堅牢性の確認**: 年度の境界（12月→1月の切り替わり等）でパスが意図せず変化しないことを確認する
- 🔴 信頼性レベル: 年度の決定方法（識別子由来か日時由来か）はタスクファイルに明記がなく、実装時に決定方針を定める必要がある推測である
