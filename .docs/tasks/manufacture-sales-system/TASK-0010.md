## タスク概要
出荷指示一覧表示・出荷完了登録（在庫の実減算）・納品書PDF出力・返品登録の機能を実装する。出荷完了時には`stock_quantity`と`reserved_quantity`を同時に減算し、在庫変動履歴（`stock_movements`）の記録とステータス更新をDBトランザクションでアトミックに実行する（dataflow.md「機能2」）。本機能は在庫・出荷担当者（warehouse）が主に利用する画面であり、請求書操作には関与しない（REQ-003）。

## 実装詳細

### 1. Shipmentモデル・ShipmentRepository・ShipmentService実装 🔵
**信頼性**: 🔵 *architecture.mdのRepository+Serviceパターン、database-schema.sql（shipmentsテーブル: sales_order_id, shipped_at, shipped_by, delivery_note_path等のカラム定義）より*
- `Shipment`モデル（`app/Models/Shipment.php`）: `salesOrder`（belongsTo SalesOrder）, `shippedBy`（belongsTo User）のリレーションを定義し、出荷完了日時（`shipped_at`）・納品書PDFパス（`delivery_note_path`）・返品関連カラム（`returned_at`, `return_reason`）を扱う
- `ShipmentRepositoryInterface`と`EloquentShipmentRepository`（`app/Repositories/`配下）を実装し、出荷指示済み受注の一覧取得（ページネーション50件 NFR-021）・出荷レコード作成・更新を担う
- `ShipmentService`（`app/Services/ShipmentService.php`）: 出荷完了処理（`complete()`）・納品書PDF生成依頼・返品処理（`processReturn()`）のユースケースを集約する
**実装ファイル**: `app/Models/Shipment.php`, `app/Repositories/ShipmentRepositoryInterface.php`, `app/Repositories/EloquentShipmentRepository.php`, `app/Services/ShipmentService.php`

### 2. ShipmentController実装（出荷指示一覧、出荷完了登録、納品書PDFダウンロード、返品登録） 🔵
**信頼性**: 🔵 *api-endpoints.md「出荷管理」セクション（GET /shipments, POST /shipments/{order}/complete, GET /shipments/{shipment}/delivery-note, POST /shipments/{shipment}/return）より、warehouse/admin権限*
- `index()`: 出荷指示一覧表示（`sales_orders.status = shipping_instructed`(=2)の受注を一覧表示。顧客名・受注番号での検索、ページネーション50件を実装する）
- `complete()`: 出荷完了登録処理（`POST /shipments/{order}/complete`、`ShipmentService::complete()`を呼び出し、完了後に納品書PDFダウンロードリンクを含む成功レスポンスを返す）
- `deliveryNote()`: 納品書PDFダウンロード（`GET /shipments/{shipment}/delivery-note`、`delivery_note_path`に保存されたPDFをダウンロードレスポンスとして返す。warehouse/sales/adminがアクセス可能）
- `processReturn()`: 返品登録処理（`POST /shipments/{shipment}/return`、返品理由を受け取り`ShipmentService::processReturn()`を呼び出す）
- ルーティングは一覧・出荷完了・返品登録を`middleware(['auth', 'role:warehouse,admin'])`、納品書ダウンロードを`role:warehouse,sales,admin`に限定する（api-endpoints.mdの権限定義と整合。warehouseは請求書操作不可 REQ-003）
**実装ファイル**: `app/Http/Controllers/ShipmentController.php`, `app/Http/Requests/CompleteShipmentRequest.php`, `app/Http/Requests/ProcessReturnRequest.php`, `routes/web.php`

### 3. 出荷完了処理: 在庫実減算(products.stock_quantity, reserved_quantity を同時に減算)、stock_movements記録(reason=shipment=3)、ステータスをshipped(3)に変更、shipped_at記録をDBトランザクションで実行 🔵
**信頼性**: 🔵 *REQ-051「出荷完了登録時、システムは在庫数を実際に減算しなければならない」、dataflow.md「機能2: 出荷完了登録（在庫の実減算）」シーケンス図（stock_quantityとreserved_quantityを同時に減算しトランザクション内で実行）、database-schema.sql（sales_orders.status: 3=出荷完了(shipped)、stock_movements.reason: 3=出荷による減算(shipment)）より*
- `ShipmentService::complete(SalesOrder $order)`を実装し、以下を単一のDBトランザクション（`DB::transaction()`）内でアトミックに実行する（dataflow.md「機能2」のシーケンス図に従う）:
  1. 対象受注のステータスが`shipping_instructed`(=2)であることを確認する
  2. 受注明細に紐づく各製品を`lockForUpdate()`で取得し、`stock_quantity`（実在庫）と`reserved_quantity`（引当）を明細数量分だけ同時に減算する（UPDATE products）
  3. `stock_movements`に在庫変動履歴を記録する（`reason=3`(shipment)、`quantity`・`operated_by`・`related_order_id`・操作者・日時を記録 REQ-072）
  4. `shipments`レコードを作成（または更新）し、`shipped_at`に現在時刻、`shipped_by`にログインユーザーIDを記録する
  5. `sales_orders.status`を`3`（shipped）に更新する
- トランザクションのコミット後、`ShipmentService`は`PdfService`（TASK-0004）に納品書PDF生成を依頼し、生成されたPDFのパスを`shipments.delivery_note_path`に保存する（dataflow.md「機能2」シーケンス図のC→S→P→Sの流れ）
- 減算後に`stock_quantity >= 0`かつ`reserved_quantity >= 0`であることをアプリケーションロジックで検証し、不整合時は例外をスローしてロールバックする
- トランザクション失敗時は全体をロールバックする
**実装ファイル**: `app/Services/ShipmentService.php`, `app/Repositories/EloquentProductRepository.php`, `app/Models/StockMovement.php`, `app/Models/Shipment.php`, `app/Http/Controllers/ShipmentController.php`

### 4. 納品書PDF生成(PdfService(TASK-0004)を活用したDeliveryNotePdfテンプレート、delivery_note_pathに保存) 🔵
**信頼性**: 🔵 *REQ-052「システムは納品書をPDF出力できなければならない」、dataflow.md「機能2」シーケンス図（S→P: 納品書PDF生成を依頼、P-->>S: PDFファイル）、database-schema.sql（shipments.delivery_note_path カラム）より*
- TASK-0004で実装される`PdfService`を利用し、納品書PDFを生成する`DeliveryNotePdfTemplate`（Bladeテンプレート、`resources/views/pdf/shipments/delivery_note.blade.php`）を作成する
- テンプレートには出荷日・受注番号・顧客情報・明細（製品名・数量・単価・金額）・出荷担当者名を含める
- 生成したPDFファイルをストレージ（`storage/app/delivery_notes/`等）に保存し、そのパスを`shipments.delivery_note_path`に記録する
- `ShipmentController::deliveryNote()`からダウンロードレスポンスとして提供する
- PDF生成失敗時はエラーメッセージを表示し再試行を促す（EDGE-003、Laravel標準の例外ハンドリングと連携）。出荷完了処理自体（在庫減算・ステータス更新）はPDF生成の成否に関わらずコミットされ、PDFは後から再生成できる設計とする
**実装ファイル**: `app/Services/ShipmentService.php`, `resources/views/pdf/shipments/delivery_note.blade.php`, `app/Http/Controllers/ShipmentController.php`

### 5. 返品処理: 在庫加算(stock_quantity加算)、stock_movements記録(reason=return_received=4)、ステータスをreturned(6)に変更、returned_at/return_reason記録 🔵
**信頼性**: 🔵 *REQ-053「返品時、システムは在庫数を加算しステータスを「返品済み」に変更しなければならない」、database-schema.sql（sales_orders.status: 6=返品済み(returned)、stock_movements.reason: 4=返品による加算(return)）より*
- `ShipmentService::processReturn(Shipment $shipment, string $returnReason)`を実装し、以下を単一のDBトランザクション内でアトミックに実行する:
  1. 対象受注のステータスが`shipped`(=3)（または請求済み等、返品が許容される状態）であることを確認する
  2. 返品対象の受注明細に紐づく各製品の`stock_quantity`を返品数量分だけ加算する（UPDATE products。返品時は引当`reserved_quantity`への影響はないものとする。実在庫のみを戻す）
  3. `stock_movements`に在庫変動履歴を記録する（`reason=4`(return_received)、`quantity`・`operated_by`・`related_order_id`を記録）
  4. `sales_orders.status`を`6`（returned）に更新し、`returned_at`に現在時刻、`return_reason`に入力された返品理由を記録する
- 返品数量が出荷数量を超えないことをバリデーションする（全量返品を基本とし、部分返品の要否は実装時に確認する）
- トランザクション失敗時は全体をロールバックする
**実装ファイル**: `app/Services/ShipmentService.php`, `app/Repositories/EloquentProductRepository.php`, `app/Models/StockMovement.php`, `app/Http/Controllers/ShipmentController.php`

## 単体テスト要件

### テストケース1: ShipmentService::complete()が在庫減算とステータス変更をアトミックに行う 🔵
**信頼性**: 🔵 *REQ-051、dataflow.md「機能2」シーケンス図（トランザクション内でstock_quantityとreserved_quantityを同時に減算し、stock_movements記録・ステータス更新を実行）より*
- **Given**: `shipping_instructed`(=2)ステータスの受注が存在し、関連製品の`stock_quantity`・`reserved_quantity`が出荷数量以上である
- **When**: `ShipmentService::complete($order)`を実行する
- **Then**: 各製品の`stock_quantity`と`reserved_quantity`が出荷数量分だけ同時に正しく減算され、`stock_movements`に`reason=3`(shipment)のレコードが記録され、`shipments`レコードが作成され`shipped_at`・`shipped_by`が記録され、`sales_orders.status`が`3`(shipped)に更新されること。これらすべてが単一のDBトランザクション内で実行され、一部でも失敗した場合は全件ロールバックされること

### テストケース2: 返品処理が在庫を正しく加算しstock_movementsに記録する 🔵
**信頼性**: 🔵 *REQ-053「返品時、システムは在庫数を加算しステータスを「返品済み」に変更しなければならない」、database-schema.sql（stock_movements.reason: 4=返品による加算）より*
- **Given**: `shipped`(=3)ステータスの出荷済み受注（`shipments`レコードあり）が存在する
- **When**: `ShipmentService::processReturn($shipment, '製品不良のため')`を実行する
- **Then**: 関連製品の`stock_quantity`が返品数量分だけ正しく加算され、`stock_movements`に`reason=4`(return_received)のレコードが記録され、`sales_orders.status`が`6`(returned)に更新され、`returned_at`に現在時刻、`return_reason`に「製品不良のため」が記録されること。これらすべてが単一のDBトランザクション内で実行されること

### テストケース3: 出荷指示が発行されていない受注は出荷完了登録できない 🟡
**信頼性**: 🟡 *dataflow.md「受注ステータス遷移」（出荷指示済み→出荷完了の遷移のみが定義されている）、REQ-050（出荷指示一覧表示）の趣旨から、出荷完了登録は出荷指示済みの受注に限定されることが妥当に推測される*
- **Given**: `confirmed`(=1)（出荷指示が未発行）の受注が存在する
- **When**: `ShipmentService::complete($order)`を実行する
- **Then**: 不正なステータス遷移を示す業務エラー（例外）が返却され、在庫情報・`stock_movements`・`sales_orders.status`のいずれにも変更が加わらないこと

### テストケース4: 在庫減算後にstock_quantity・reserved_quantityが負値にならないことを保証する 🔵
**信頼性**: 🔵 *dataflow.md「データ整合性の保証」（利用可能在庫がマイナスにならないことをDB制約とアプリケーションロジックの両方で保証する）より*
- **Given**: 何らかの理由で受注明細数量が対象製品の`stock_quantity`または`reserved_quantity`を上回っている不整合な状態（テスト用に意図的に作成）
- **When**: `ShipmentService::complete($order)`を実行する
- **Then**: 整合性エラーを示す例外がスローされ、トランザクションがロールバックされ、在庫数が負値にならないこと

## 注意事項
- 出荷完了処理（実在庫・引当在庫の同時減算を伴う）は必ずDBトランザクション（`DB::transaction()`）内で実行し、対象製品レコードには`lockForUpdate()`による悲観的ロックを適用すること（dataflow.md「データ整合性の保証」と整合）
- `stock_quantity >= 0`かつ`reserved_quantity >= 0`の整合性が常に保たれることを、アプリケーションロジックとDB制約（CHECK制約）の両面で保証すること
- 納品書PDF生成はトランザクションの外（コミット後）で実行し、PDF生成に失敗しても在庫減算・ステータス更新自体はロールバックされない設計とすることを推奨する（PDFは後から再生成可能とし、業務上重要な在庫整合性を優先する）。この設計方針は実装時にチームと確認すること
- 返品時の部分返品の可否（出荷数量の一部のみを返品できるか）は要件に明記がないため、実装時に確認すること。本タスクでは全量返品を基本方針として設計している
- 本機能はwarehouse権限のユーザーが主に利用するため、請求書関連の操作・表示が一切含まれないことをルーティング・コントローラ・ビューの全層で確認すること（REQ-003）
