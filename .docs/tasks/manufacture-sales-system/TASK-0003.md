## タスク概要
Laravel Breezeの認証scaffoldをベースに、本システム固有の4ロール（システム管理者・営業担当者・在庫出荷担当者・管理職経理担当）に応じたアクセス制御基盤を実装する。`users.role`カラムに対応した認証フローのカスタマイズ、ロールベースアクセス制御を行う`EnsureUserHasRole`ミドルウェア、Gate/Policyによる権限制御基盤、パスワードリセット機能、セッションタイムアウト設定までを構築し、後続の業務機能タスク（TASK-0004, 0005, 0006, 0007）が利用するアクセス制御の土台を整備する。

## 実装詳細

### 1. Breeze認証フローのカスタマイズ（roleカラム対応・ロール別リダイレクト） 🔵
**信頼性**: 🔵 *requirements.md REQ-001（メール・パスワードログイン）, REQ-002（4ロール）、database-schema.sql users.roleカラム定義より*
- `LoginController`（または`AuthenticatedSessionController`）のログイン成功処理をカスタマイズし、ログイン後のリダイレクト先を`User::role`（`UserRole` Enum）に応じて分岐させる（例: admin→管理ダッシュボード、sales→見積/受注一覧、warehouse→出荷指示一覧、accounting→請求/入金管理画面）
- `User`モデルに`role`を`UserRole`にキャストする属性キャスト（`casts()`に`'role' => UserRole::class`）を追加し、`hasRole(UserRole $role): bool`等のヘルパーメソッドを実装する
- `is_active = false`（無効化済み）のユーザーはログインを拒否し、適切なエラーメッセージを表示する（REQ-004の無効化機能との連携）
**実装ファイル**: `app/Http/Controllers/Auth/AuthenticatedSessionController.php`, `app/Models/User.php`, `app/Providers/RouteServiceProvider.php`（または相当のリダイレクトロジック）

### 2. roleミドルウェア(EnsureUserHasRole)実装、ルートグループへの適用 🔵
**信頼性**: 🔵 *architecture.md「ミドルウェア: 認証（auth）、役割ベースアクセス制御（role middleware）」、ディレクトリ構造`app/Http/Middleware/EnsureUserHasRole.php`より*
- `EnsureUserHasRole`ミドルウェアを実装し、`handle(Request $request, Closure $next, string ...$roles)`で許可ロールを可変引数として受け取る
- ログインユーザーの`role`が許可ロール一覧に含まれない場合、`abort(403)`で拒否する
- `bootstrap/app.php`（Laravel 13のミドルウェアエイリアス登録方式）で`role`エイリアスとして登録し、`routes/web.php`の各ルートグループに`->middleware(['auth', 'role:admin,accounting'])`のように適用する
**実装ファイル**: `app/Http/Middleware/EnsureUserHasRole.php`, `bootstrap/app.php`, `routes/web.php`

### 3. Gate/Policyによる役割別アクセス制御基盤実装 🔵
**信頼性**: 🔵 *requirements.md REQ-003（在庫・出荷担当者は請求書操作不可）, REQ-064（管理職・経理担当者のみ請求書発行・入金確認可）、architecture.md「Gate/Policyによる役割ベースアクセス制御」より*
- `AuthServiceProvider`（または`bootstrap/app.php`の`Gate::define`）に基本的なGateを定義する（例: `Gate::define('manage-invoices', fn(User $user) => in_array($user->role, [UserRole::ACCOUNTING, UserRole::ADMIN]))`）
- `InvoicePolicy`（`app/Policies/InvoicePolicy.php`）の雛形を作成し、`view`/`create`/`update`等のアビリティをaccounting/adminロールに限定する基盤を用意する（具体的な請求書機能本体は後続タスクで実装するが、本タスクではPolicyの基盤・登録の仕組みを構築する）
- Controllerやルートから`$this->authorize()`、`Gate::allows()`、Bladeの`@can`ディレクティブで利用できる状態にする
**実装ファイル**: `app/Providers/AuthServiceProvider.php`（または`bootstrap/app.php`）, `app/Policies/InvoicePolicy.php`

### 4. パスワードリセット機能（Breeze標準活用） 🟡
**信頼性**: 🟡 *requirements.md REQ-005「システムはパスワードリセット機能を提供しなければならない」（🟡認証機能から妥当な推測）より、Breeze標準機能をそのまま活用する設計*
- Breeze標準の`ForgotPasswordController`/`ResetPasswordController`相当の機能（`routes/auth.php`に生成済み）をそのまま活用し、本システムのメール送信設定（`.env`の`MAIL_*`）と連携させる
- パスワードリセットメールのBladeテンプレートを必要に応じて日本語化・本システム向けにカスタマイズする
- リセット後のパスワードもbcryptでハッシュ化されることを確認する（NFR-010）
**実装ファイル**: `routes/auth.php`, `app/Http/Controllers/Auth/PasswordResetLinkController.php`, `app/Http/Controllers/Auth/NewPasswordController.php`, `resources/views/auth/forgot-password.blade.php`, `resources/views/auth/reset-password.blade.php`

### 5. セッションタイムアウト60分の設定 🟡
**信頼性**: 🟡 *requirements.md NFR-011「セッションタイムアウト（60分）を設定しなければならない」（🟡セキュリティ要件から推測）より*
- `config/session.php`の`lifetime`を`60`（分）に設定し、`.env`の`SESSION_LIFETIME=60`を設定する
- `expire_on_close`の設定値を確認し、業務システムとしての利用形態（長時間の入力作業等）を考慮した妥当な設定とする
- セッションタイムアウト後にアクセスした場合、ログイン画面にリダイレクトされ、再ログインを促すメッセージが表示されることを確認する
**実装ファイル**: `config/session.php`, `.env`

## 単体テスト要件

### テストケース1: roleミドルウェアが権限外アクセスを403で拒否する 🔵
**信頼性**: 🔵 *requirements.md REQ-003「在庫・出荷担当者が請求書操作を実行しようとした場合、システムはアクセスを拒否しなければならない」より*
- **Given**: `role:accounting,admin`が適用されたルートに対し、`role=warehouse`（在庫・出荷担当者）でログインしたユーザーがいる
- **When**: 当該ルートへアクセスする
- **Then**: HTTPステータス403（Forbidden）が返却され、画面遷移が拒否されること

### テストケース2: 各役割でログインし適切にリダイレクトされる 🔵
**信頼性**: 🔵 *requirements.md REQ-001, REQ-002（4ロール）、architecture.md 認証方式より*
- **Given**: 4つの役割（admin, sales, warehouse, accounting）のユーザーがそれぞれ登録されている
- **When**: 各ユーザーでログインを行う
- **Then**: ログインに成功し、各役割に対応した初期画面（ダッシュボード/業務一覧）へリダイレクトされること

### テストケース3: 無効化されたユーザーはログインできない 🟡
**信頼性**: 🟡 *requirements.md REQ-004「システム管理者はユーザーの作成・編集・無効化ができなければならない」（🟡から派生する妥当な推測）より*
- **Given**: `is_active = false`のユーザーが存在する
- **When**: 当該ユーザーがメールアドレス・パスワードでログインを試みる
- **Then**: ログインが拒否され、「アカウントが無効化されています」等のエラーメッセージが表示されること

### テストケース4: パスワードがbcryptでハッシュ化されて保存される 🔵
**信頼性**: 🔵 *requirements.md NFR-010「パスワードはbcryptでハッシュ化して保存しなければならない」より*
- **Given**: 新規ユーザー登録またはパスワードリセットが行われる
- **When**: ユーザーをDBから取得し`password`カラムの値を確認する
- **Then**: 平文ではなくbcrypt形式（`$2y$`で始まるハッシュ値）で保存されていること

### テストケース5: セッションタイムアウトが60分に設定されている 🟡
**信頼性**: 🟡 *requirements.md NFR-011（セッションタイムアウト60分）より*
- **Given**: `config/session.php`の設定が読み込まれている
- **When**: `config('session.lifetime')`を取得する
- **Then**: 値が`60`であること

## 注意事項
- ロールの権限判定ロジックは`EnsureUserHasRole`ミドルウェアおよびGate/Policyに集約し、Controller内でのロール判定の重複実装を避けること（保守性確保のため）
- 権限外アクセス時は必ずHTTP 403を返却すること（REQ-003）。404にすり替えてリソースの存在を隠す設計は本要件では採用しない
- セッションタイムアウトの設定変更は、Laravelのセッションドライバ設定（`SESSION_DRIVER`）と整合性をとること
- パスワードのハッシュ化はLaravel標準（`Hash::make`、`bcrypt`ドライバ）に委ね、独自実装は行わないこと（NFR-010）
- TASK-0002で実装した`UserRole` Enumを`User`モデルの属性キャストとして利用し、ロール判定を型安全に行うこと
