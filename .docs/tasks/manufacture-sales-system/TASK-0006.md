## タスク概要
製品マスタのCRUD機能（一覧・登録・編集・検索・在庫アラート表示）と、在庫の手動調整機能を実装する。`products`テーブルが持つ実在庫数（`stock_quantity`）と引当数（`reserved_quantity`）の整合性（`reserved_quantity <= stock_quantity`）を維持する設計とし、在庫変動はすべて`stock_movements`テーブルに記録する。Repository+Serviceパターンに従い、`ProductService`に在庫計算・調整ロジックを集約する。

## 実装詳細

### 1. Productモデル・ProductRepository・ProductService実装（availableQuantity()計算ロジック含む） 🔵
**信頼性**: 🔵 *data-types.php（ProductData::availableQuantity() = stockQuantity - reservedQuantity）、database-schema.sql（products: stock_quantity, reserved_quantity, chk_products_reserved_le_stock制約）、architecture.md（Repository+Serviceパターン）より*
- `Product`モデル: `products`テーブルに対応するEloquentモデル。`product_code`, `product_name`, `unit_price`, `unit`, `stock_quantity`, `reserved_quantity`, `alert_threshold`を`$fillable`に設定。`stockMovements()`リレーション（hasMany）を定義する
- `ProductRepositoryInterface`（`app/Repositories/Contracts/`）: `paginate()`, `find()`, `create()`, `update()`, `search()`, `adjustStock()`等のメソッドを定義
- `ProductRepository`（`app/Repositories/Eloquent/`）: インターフェースのEloquent実装
- `ProductService`: 業務ロジックを集約。`availableQuantity(Product $product): int`メソッドで`stock_quantity - reserved_quantity`を計算し、見積・受注確定時の在庫チェック等から再利用可能にする
- `ProductData` DTO（既にarchitecture.mdに定義済み、`availableQuantity()`メソッドを内包）を用いてController⇔Service間のデータ受け渡しを行う
**実装ファイル**: `app/Models/Product.php`, `app/Repositories/Contracts/ProductRepositoryInterface.php`, `app/Repositories/Eloquent/ProductRepository.php`, `app/Services/ProductService.php`

### 2. ProductController実装（一覧・登録・編集・検索・在庫アラート表示） 🔵
**信頼性**: 🔵 *api-endpoints.md（GET /products, GET /products/create, POST /products, GET /products/{product}/edit, PUT /products/{product}）、REQ-020, REQ-021, REQ-022より*
- `index()`: 製品一覧を取得し、`paginate(50)`でページネーション表示する（NFR-021）。検索キーワード（品番・製品名）がある場合は検索結果を表示し、`alert_threshold`を下回る製品には警告フラグを付与する。アクセス可能ロールは全役割
- `create()` / `store()`: 新規登録フォーム表示・登録処理（バリデーション含む）。アクセス可能ロールは admin のみ
- `edit()` / `update()`: 編集フォーム表示・更新処理。アクセス可能ロールは admin のみ
- ロールごとのアクセス制御は`EnsureUserHasRole`ミドルウェアを利用する
**実装ファイル**: `app/Http/Controllers/ProductController.php`, `app/Http/Requests/StoreProductRequest.php`, `app/Http/Requests/UpdateProductRequest.php`, `routes/web.php`

### 3. 在庫手動調整機能(adjust-stock)実装、調整時にstock_movementsへ記録 🔵
**信頼性**: 🔵 *REQ-023「システムは製品の在庫数を直接増減できなければならない（入出荷による自動増減とは別に手動調整）」、REQ-072「在庫数の変動履歴（日時・変動理由・数量・操作者）を記録」、database-schema.sql（区分値コード表: stock_movements.reason 5=手動調整(manual)、stock_movementsテーブル定義）、api-endpoints.md（POST /products/{product}/adjust-stock）より*
- `POST /products/{product}/adjust-stock`エンドポイントを実装する。リクエストには増減数（`quantity_change`、正負いずれも可）とメモ（任意）を含める
- `ProductService::adjustStock(int $productId, int $quantityChange, int $operatedBy, ?string $memo)`:
  1. トランザクション内で`Product::stock_quantity`を増減し、結果が0未満や`reserved_quantity`を下回らないことを検証する（DBの`chk_products_stock`, `chk_products_reserved_le_stock`制約とアプリ側バリデーションの二重チェック）
  2. `stock_movements`テーブルへレコードを作成: `product_id`, `reason=5`（manual_adjustment, コード表参照）, `quantity_change`, `related_order_id=NULL`, `operated_by`, `memo`, `created_at`
  3. 更新後の在庫数を返却する
- アクセス可能ロールは warehouse, admin
**実装ファイル**: `app/Http/Controllers/ProductController.php`, `app/Services/ProductService.php`, `app/Http/Requests/AdjustStockRequest.php`, `app/Models/StockMovement.php`

### 4. 在庫アラート閾値(alert_threshold)による警告表示ロジック 🟡
**信頼性**: 🟡 *REQ-022「在庫数が設定した閾値を下回った場合、システムは警告を表示しなければならない」、database-schema.sql（products.alert_threshold: 🟡 在庫アラート閾値）より、表示方法（バッジ・色分け等）はマスタ管理UIの一般的UXから推測*
- `ProductService`または`ProductData`に`isLowStock(): bool`相当のロジックを設け、`stock_quantity < alert_threshold`の場合に警告フラグを立てる
- `index()`画面で警告対象の製品に対し視覚的な警告表示（バッジ・行のハイライト等）を行う
- `alert_threshold`は製品登録・編集フォームから個別に設定可能とする
**実装ファイル**: `app/Services/ProductService.php`, `resources/views/products/index.blade.php`

## 単体テスト要件

### テストケース1: ProductService::adjustStock()が在庫を増減しstock_movementsに記録する 🔵
**信頼性**: 🔵 *REQ-023, REQ-072, database-schema.sql（stock_movements.reason=5 手動調整）より*
- **Given**: `stock_quantity=100`, `reserved_quantity=20`の製品が存在する
- **When**: `ProductService::adjustStock($productId, +10, $operatedBy, 'メモ')`を呼び出す
- **Then**: 製品の`stock_quantity`が110に更新され、`stock_movements`に`reason=5`, `quantity_change=+10`, `operated_by`, `memo`を含むレコードが1件作成されること

### テストケース2: 在庫数が閾値を下回ると警告フラグが立つ 🟡
**信頼性**: 🟡 *REQ-022より*
- **Given**: `alert_threshold=10`に設定された製品の`stock_quantity`が`5`である
- **When**: 製品データを取得し警告判定ロジックを評価する
- **Then**: 当該製品の警告フラグが`true`となり、一覧画面で警告表示の対象になること

### テストケース3: 在庫調整により実在庫が引当数を下回る操作が拒否される 🔵
**信頼性**: 🔵 *database-schema.sql（chk_products_reserved_le_stock制約: reserved_quantity <= stock_quantity）より*
- **Given**: `stock_quantity=20`, `reserved_quantity=15`の製品が存在する
- **When**: `ProductService::adjustStock($productId, -10, $operatedBy)`（結果が`stock_quantity=10 < reserved_quantity=15`となる）を呼び出す
- **Then**: バリデーションエラー（例外）がスローされ、在庫は更新されず、`stock_movements`にもレコードが作成されないこと

### テストケース4: availableQuantity()が実在庫と引当数の差分を正しく返す 🔵
**信頼性**: 🔵 *data-types.php（ProductData::availableQuantity() = stockQuantity - reservedQuantity）より*
- **Given**: `stock_quantity=50`, `reserved_quantity=12`の製品が存在する
- **When**: `availableQuantity()`を呼び出す
- **Then**: 戻り値が`38`であること

## 注意事項
- 在庫数の更新は必ずトランザクション内で行い、`stock_movements`への記録と`products.stock_quantity`の更新が必ず一貫して成功/失敗するようにすること（整合性確保）
- DBレベルの制約（`chk_products_stock`, `chk_products_reserved`, `chk_products_reserved_le_stock`）に依存しすぎず、アプリケーション層でも事前検証を行いユーザーに分かりやすいエラーを返すこと
- `stock_movements.reason`の値はコード表（database-schema.sql記載）と完全に一致させ、Enum（`StockMovementReason`等）で管理することを推奨する
- 受注引当（reason=1）・引当解除（reason=2）・出荷減算（reason=3）・返品加算（reason=4）は別タスク（受注・出荷機能）で実装されるため、本タスクでは手動調整（reason=5）のみを対象とする
- 検索処理はSQLインジェクション対策としてEloquent ORMのクエリビルダを使用すること（NFR-013）
