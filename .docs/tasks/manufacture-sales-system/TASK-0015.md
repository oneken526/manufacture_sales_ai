## タスク概要
jQuery AJAX用の内部APIエンドポイント（`/api/internal/*`）を実装する。顧客検索・製品検索のインクリメンタルサーチ、受注確定前の利用可能在庫チェック、見積金額のリアルタイム計算という、各画面（顧客検索、見積作成画面）のUXを支える横断的な基盤機能を提供する。レスポンスはJSON形式とし、共通のエラーレスポンス形式（`success`, `error.code`, `error.message`, `error.details`）に統一する。

## 実装詳細

### 1. 内部APIコントローラ群の実装 🔵
**信頼性**: 🔵 *api-endpoints.md「内部APIルート（jQuery AJAX用 / `/api/internal/*`）」より*
- `app/Http/Controllers/Api/CustomerSearchController.php`: 顧客検索API（`search()`）を提供。既存の`CustomerService`/`CustomerRepository::search()`（TASK-0005）を呼び出す
- `app/Http/Controllers/Api/ProductSearchController.php`: 製品検索API（`search()`）を提供。既存の`ProductService`/`ProductRepository::search()`（TASK-0006）を呼び出す
- `app/Http/Controllers/Api/StockAvailabilityController.php`: 利用可能在庫チェックAPI（`show()`）を提供。`ProductData::availableQuantity()`（既にdata-types.phpに定義済み: `stockQuantity - reservedQuantity`）を利用する
- `app/Http/Controllers/Api/QuotationCalculationController.php`: 見積金額のリアルタイム計算API（`calculate()`）を提供。既存の見積金額計算ロジック（TASK-0008の`QuotationService`）を再利用または共通化する
- 全コントローラはセッションベース認証（Laravel Breeze）配下に置き、`routes/api.php`の`/api/internal/*`グループに登録する
**実装ファイル**: `app/Http/Controllers/Api/CustomerSearchController.php`, `app/Http/Controllers/Api/ProductSearchController.php`, `app/Http/Controllers/Api/StockAvailabilityController.php`, `app/Http/Controllers/Api/QuotationCalculationController.php`, `routes/api.php`

### 2. 顧客・製品の部分一致インクリメンタル検索API 🔵
**信頼性**: 🔵 *REQ-011「システムは顧客を検索（会社名・担当者名・電話番号）できなければならない」、REQ-021「システムは製品を検索（品番・製品名）できなければならない」、api-endpoints.md（GET /api/internal/customers/search?q={keyword}, GET /api/internal/products/search?q={keyword}）より*
- `GET /api/internal/customers/search?q={keyword}`: クエリパラメータ`q`を受け取り、`CustomerRepository::search($keyword)`（会社名・担当者名・電話番号へのLIKE部分一致OR検索、TASK-0005で実装済み）の結果を`{ id, companyName, contactName, phone }`等の軽量なJSON配列で返す
- `GET /api/internal/products/search?q={keyword}`: クエリパラメータ`q`を受け取り、`ProductRepository::search($keyword)`（品番・製品名へのLIKE部分一致OR検索、TASK-0006で実装済み）の結果を`{ id, productCode, productName, unitPrice, availableQuantity }`等の軽量なJSON配列で返す
- `q`が空または未指定の場合は空配列を返す（全件取得を防止しパフォーマンスに配慮、NFR-001）
- レスポンス件数は一定数（例: 上位20件）に制限し、オートコンプリート用途として過剰なデータ転送を防ぐ
**実装ファイル**: `app/Http/Controllers/Api/CustomerSearchController.php`, `app/Http/Controllers/Api/ProductSearchController.php`, `app/Http/Resources/CustomerSearchResource.php`, `app/Http/Resources/ProductSearchResource.php`

### 3. 利用可能在庫チェックAPI 🔵
**信頼性**: 🔵 *EDGE-001「在庫数以上の数量を受注しようとした場合、システムは警告を表示して処理を中止しなければならない」、EDGE-010「在庫数が0の場合、製品の受注登録はできないものとする（警告表示）」、api-endpoints.mdレスポンス例（stockQuantity, reservedQuantity, availableQuantity, sufficient）より*
- `GET /api/internal/products/{product}/availability?quantity={n}`: 指定製品の`stockQuantity`（実在庫数）, `reservedQuantity`（引当中在庫数）, `availableQuantity`（`stockQuantity - reservedQuantity`、`ProductData::availableQuantity()`を利用）, `sufficient`（`availableQuantity >= quantity`の真偽値）を含むJSONを返す
- レスポンス形式はapi-endpoints.mdのレスポンス例に準拠する: `{ "success": true, "data": { "productId": 12, "stockQuantity": 50, "reservedQuantity": 20, "availableQuantity": 30, "sufficient": true } }`
- `quantity`パラメータが未指定または不正な値の場合はバリデーションエラー（共通エラーレスポンス形式）を返す
- 受注確定画面・見積作成画面からのリアルタイムバリデーション（EDGE-001, EDGE-010）に利用される
**実装ファイル**: `app/Http/Controllers/Api/StockAvailabilityController.php`, `app/Http/Resources/StockAvailabilityResource.php`

### 4. 見積金額のリアルタイム計算API 🟡
**信頼性**: 🟡 *api-endpoints.md「見積・受注の動的明細操作」（POST /api/internal/quotations/calculate、🟡見積作成のUX（行追加・削除）から妥当な推測）より*
- `POST /api/internal/quotations/calculate`: リクエストボディに明細行配列（`{ productId, quantity, unitPrice }[]`）を受け取り、各行の小計（`quantity * unitPrice`）と見積全体の合計金額を計算して返す
- レスポンスは`{ success: true, data: { items: [{ productId, quantity, unitPrice, subtotal }], totalAmount } }`形式とする
- `QuotationItemData` DTO（既にdata-types.phpに定義済み: `productId`, `quantity`, `unitPrice`）の構造に準拠したリクエスト・レスポンス設計とする
- 見積作成画面で行追加・数量変更・単価変更のたびにこのAPIを呼び出し、合計金額をリアルタイム表示する想定
- 計算ロジックはTASK-0008の`QuotationService`内の金額計算処理と重複させず、共通の計算クラス/トレイトとして切り出して再利用することが望ましい
**実装ファイル**: `app/Http/Controllers/Api/QuotationCalculationController.php`, `app/Http/Requests/CalculateQuotationRequest.php`, `app/Services/QuotationCalculationService.php`

### 5. JSON共通エラーレスポンス形式の統一実装 🟡
**信頼性**: 🟡 *api-endpoints.md「共通エラーレスポンス（内部API）」（🟡既存実装の共通パターンから妥当な推測、`{ success, error: { code, message, details } }`形式）より*
- 全内部APIエンドポイントで共通のJSONエラーレスポンス形式`{ "success": false, "error": { "code": "...", "message": "...", "details": {...} } }`を使用する
- `app/Exceptions/Handler.php`（またはAPI専用の例外ハンドラ）で、`/api/internal/*`配下のリクエストに対する例外を共通形式のJSONに変換するロジックを実装する
- バリデーションエラー（422）、未認証（401）、権限外アクセス（403）、リソース未検出（404）、業務エラー（400/409等、例: `STOCK_INSUFFICIENT`）をそれぞれ適切な`error.code`にマッピングする
- `error.details`には、在庫不足時の`{ productId, available, requested }`等、クライアント側での表示・ハンドリングに必要な補足情報を含める
**実装ファイル**: `app/Exceptions/Handler.php`, `app/Http/Responses/ApiErrorResponse.php`

## 単体テスト要件

### テストケース1: 在庫チェックAPIが在庫充足/不足を正しく判定する 🔵
**信頼性**: 🔵 *EDGE-001, EDGE-010より*
- **Given**: 実在庫50・引当中在庫20（利用可能在庫30）の製品が登録されている
- **When**: `GET /api/internal/products/{product}/availability?quantity=10`および`?quantity=50`を呼び出す
- **Then**: 前者は`sufficient: true`、後者は`sufficient: false`を返し、いずれも`stockQuantity`, `reservedQuantity`, `availableQuantity`が正しい値で含まれること

### テストケース2: 在庫が0の製品は常に不足と判定される 🔵
**信頼性**: 🔵 *EDGE-010「在庫数が0の場合、製品の受注登録はできないものとする」より*
- **Given**: 実在庫0の製品が登録されている
- **When**: `GET /api/internal/products/{product}/availability?quantity=1`を呼び出す
- **Then**: `availableQuantity: 0`, `sufficient: false`を返すこと

### テストケース3: 顧客検索APIが部分一致検索結果をJSON配列で返す 🟡
**信頼性**: 🟡 *REQ-011、api-endpoints.mdの内部API検索エンドポイント定義より*
- **Given**: 会社名・担当者名・電話番号が異なる複数の顧客レコードが登録されている
- **When**: `GET /api/internal/customers/search?q={キーワード}`を呼び出す
- **Then**: 会社名・担当者名・電話番号のいずれかにキーワードを部分一致で含む顧客のみがJSON配列として返されること。該当なしの場合は空配列が返ること

### テストケース4: 製品検索APIが部分一致検索結果をJSON配列で返す 🟡
**信頼性**: 🟡 *REQ-021、api-endpoints.mdの内部API検索エンドポイント定義より*
- **Given**: 品番・製品名が異なる複数の製品レコードが登録されている
- **When**: `GET /api/internal/products/search?q={キーワード}`を呼び出す
- **Then**: 品番・製品名のいずれかにキーワードを部分一致で含む製品のみがJSON配列として返されること

### テストケース5: 見積金額計算APIが明細行から正しい小計・合計を計算する 🟡
**信頼性**: 🟡 *api-endpoints.md「見積・受注の動的明細操作」（🟡）、QuotationItemData DTOより*
- **Given**: 複数の明細行（製品・数量・単価の組み合わせ）を含むリクエストボディが用意されている
- **When**: `POST /api/internal/quotations/calculate`を呼び出す
- **Then**: 各行の小計（`quantity * unitPrice`）と全体の合計金額が正しく計算されてレスポンスに含まれること

### テストケース6: 共通エラーレスポンス形式が統一されている 🟡
**信頼性**: 🟡 *api-endpoints.md「共通エラーレスポンス（内部API）」より*
- **Given**: 存在しない製品IDで在庫チェックAPIを呼び出す、またはバリデーションエラーとなるリクエストを送信する
- **When**: 各内部APIエンドポイントを呼び出す
- **Then**: レスポンスが`{ success: false, error: { code, message, details } }`形式で統一されており、HTTPステータスコードも適切（404, 422等）であること

## 注意事項
- 内部APIは`/api/internal/*`としてバージョニングを行わない（api-endpoints.md「APIバージョニング」より）。外部公開APIは別途`/api/v1/`形式で検討する
- 検索処理はSQLインジェクション対策としてEloquent ORMのクエリビルダを使用すること（NFR-013）。既存のCustomerRepository::search()/ProductRepository::search()（TASK-0005, TASK-0006）を再利用し、検索ロジックの重複実装を避けること
- 在庫チェックAPIと見積金額計算APIは、受注確定・見積保存時のサーバーサイドの最終バリデーションとは別物であり、あくまでUX向上のためのリアルタイムフィードバックである点に注意する（最終的な整合性チェックは各機能タスクのService層で再度行う）
- レスポンスサイズと検索パフォーマンスに配慮し、検索結果の返却件数を制限する（オートコンプリート用途として過剰なデータ転送を避ける、NFR-001）
- セッションベース認証配下のため、CSRFトークンをAjaxリクエストヘッダ（`X-CSRF-TOKEN`）に含める実装をフロントエンド（jQuery）側で行うこと
