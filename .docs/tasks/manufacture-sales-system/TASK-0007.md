## タスク概要
システム管理者（admin）専用のユーザー管理画面を実装する。ユーザーの新規作成・編集・無効化（`is_active`フラグ）を行い、無効化されたユーザーはログインできないように制御する。また、Laravel Breeze標準のパスワードリセット機能を活用し、管理者またはユーザー自身がパスワードをリセットできるフローを整備する。

## 実装詳細

### 1. UserController(管理者専用)実装：ユーザー作成・編集・無効化(is_activeフラグ) 🟡
**信頼性**: 🟡 *REQ-004「システム管理者はユーザーの作成・編集・無効化ができなければならない」から妥当な推測。database-schema.sql（users.is_active: 🟡 REQ-004より無効化機能から推測）と整合*
- `UserController`（`app/Http/Controllers/`または`app/Http/Controllers/Admin/`配下）を実装する
- `index()`: ユーザー一覧表示（名前・メールアドレス・役割・有効状態を表示、ページネーション50件 NFR-021）
- `create()` / `store()`: ユーザー新規作成フォーム表示・登録処理（名前・メールアドレス・初期パスワード・役割の入力、バリデーション含む）
- `edit()` / `update()`: ユーザー編集フォーム表示・更新処理（名前・メールアドレス・役割の変更）
- `toggleActive()`（または`deactivate()`/`activate()`）: `is_active`フラグの切り替え処理。無効化時は確認ダイアログを経由する
- ルーティングは`middleware(['auth', 'role:admin'])`等、`EnsureUserHasRole`ミドルウェアでadminロールに限定する（REQ-002, REQ-003の役割管理方針と整合）
**実装ファイル**: `app/Http/Controllers/Admin/UserController.php`, `app/Http/Requests/StoreUserRequest.php`, `app/Http/Requests/UpdateUserRequest.php`, `routes/web.php`

### 2. is_activeフラグによるログイン制御(無効化ユーザーのログイン拒否) 🟡
**信頼性**: 🟡 *REQ-004（無効化機能）から、無効化されたユーザーはログインできないという挙動が妥当に推測される。database-schema.sql（users.is_active BOOLEAN NOT NULL DEFAULT TRUE）と整合*
- Laravel認証フロー（Breeze標準のログイン処理）に対し、`is_active=false`のユーザーの認証を拒否するロジックを追加する
- 実装方法としては、認証イベントリスナー（`Illuminate\Auth\Events\Attempting`/`Login`）でのチェック、またはカスタム`Authenticatable`実装やログインリクエストでの追加検証（`LoginRequest::authenticate()`オーバーライド）のいずれかを用いる
- 拒否時はログイン画面に「アカウントが無効化されています。管理者にお問い合わせください」等のメッセージを表示し、セッションを開始しない
**実装ファイル**: `app/Http/Requests/Auth/LoginRequest.php`（または`app/Listeners/`配下のリスナー）, `app/Providers/EventServiceProvider.php`

### 3. パスワードリセットメール送信フロー(Breeze標準機能の活用) 🟡
**信頼性**: 🟡 *REQ-005「システムはパスワードリセット機能を提供しなければならない」から、Laravel Breezeに標準搭載されたパスワードリセット機能の活用が妥当に推測される*
- Laravel Breezeに標準で含まれる`ForgotPasswordController`/`ResetPasswordController`相当の機能（`Password`ファサード、`PasswordReset`通知）をそのまま活用する
- メール送信は`.env`のメール設定（開発環境では`MAIL_MAILER=log`等）を用いて検証する
- 管理者が任意のユーザーに対してパスワードリセットメールを再送できる導線（任意機能）をユーザー管理画面に設けることを検討する
- 無効化されたユーザー（`is_active=false`）に対してはパスワードリセットを許可しない、またはリセット後も引き続きログイン拒否される設計とする
**実装ファイル**: `app/Http/Controllers/Auth/PasswordResetLinkController.php`（Breeze標準）, `resources/views/auth/forgot-password.blade.php`（Breeze標準）, `app/Http/Controllers/Admin/UserController.php`

## 単体テスト要件

### テストケース1: 無効化されたユーザーがログインできない 🟡
**信頼性**: 🟡 *REQ-004から妥当な推測*
- **Given**: `is_active=false`に設定されたユーザーアカウントが存在する
- **When**: 当該ユーザーが正しいメールアドレスとパスワードでログインを試みる
- **Then**: 認証が拒否され、「アカウントが無効化されています」等のエラーメッセージが返却され、セッションが開始されないこと

### テストケース2: admin以外がユーザー管理画面にアクセスできない 🔵
**信頼性**: 🔵 *REQ-002「システムは4つの役割（システム管理者・営業担当者・在庫出荷担当者・管理職経理担当）を持たなければならない」、REQ-003（役割に応じたアクセス拒否方針）より*
- **Given**: sales, warehouse, accountingいずれかのロールを持つユーザーがログインしている
- **When**: ユーザー管理画面（`/admin/users`等）へアクセスを試みる
- **Then**: アクセスが拒否され、403エラーまたは適切なリダイレクト・エラーメッセージが返されること

### テストケース3: 有効なユーザーが正常にログインできる 🟡
**信頼性**: 🟡 *REQ-001（メールアドレスとパスワードによるログイン）、REQ-004（is_activeフラグ導入による回帰確認）より*
- **Given**: `is_active=true`に設定されたユーザーアカウントが存在する
- **When**: 正しいメールアドレスとパスワードでログインを試みる
- **Then**: 認証が成功し、ダッシュボード等の認証後画面へ遷移すること

## 統合テスト要件

### 統合テスト1: ユーザー作成→無効化→ログイン拒否確認の一連フロー 🟡
**信頼性**: 🟡 *REQ-004, REQ-005より*
- **シナリオ**:
  1. admin権限のユーザーでログインし、ユーザー管理画面から新規ユーザー（役割: sales）を作成する
  2. 作成したユーザーでログインできることを確認する
  3. admin権限のユーザーが当該ユーザーを無効化（`is_active=false`）する（無効化確認ダイアログを経由）
  4. 無効化されたユーザーが再度ログインを試み、拒否されることを確認する
- **期待結果**: 一連の操作がエラーなく完了し、無効化フラグがログイン制御に正しく反映されること

### 統合テスト2: パスワードリセットフロー 🟡
**信頼性**: 🟡 *REQ-005より*
- **シナリオ**:
  1. ログイン画面の「パスワードを忘れた場合」リンクからパスワードリセット申請を行う
  2. リセットメールが送信される（テスト環境ではログ出力等で検証）
  3. メール内のリンクから新しいパスワードを設定する
  4. 新しいパスワードでログインできることを確認する
- **期待結果**: Breeze標準のパスワードリセットフローが問題なく機能すること

## UI/UX要件

### ローディング状態 🟡
**信頼性**: 🟡 *NFR-001（3秒以内のページ表示）より*
- ユーザー作成・編集フォーム送信中はボタンを無効化し「処理中...」表示を行う
- 無効化処理実行中はインジケータを表示する

### エラー表示 🟡
**信頼性**: 🟡 *Laravelバリデーション標準機能より*
- ユーザー作成・編集フォームでバリデーションエラー（メールアドレス重複、パスワード強度不足等）が発生した場合、各入力項目の直下にエラーメッセージを表示する
- 無効化されたユーザーのログイン試行時はログイン画面にエラーメッセージを表示する

### 役割選択ドロップダウン 🟡
**信頼性**: 🟡 *REQ-002（4つの役割: システム管理者・営業担当者・在庫出荷担当者・管理職経理担当）より、UI上の選択方式は一般的なマスタ管理UXから推測*
- ユーザー作成・編集フォームに役割選択用のドロップダウン（セレクトボックス）を設置し、4つの役割（admin/sales/warehouse/accounting）を選択肢として表示する
- 選択中の役割に応じた説明テキスト（例:「在庫・出荷担当者は請求書操作ができません」REQ-003）を補足表示することを検討する

### 無効化確認ダイアログ 🟡
**信頼性**: 🟡 *REQ-004（無効化機能）より、誤操作防止の観点から確認ダイアログの設置が妥当に推測される*
- ユーザーを無効化する操作の前に、JavaScript（confirm()またはモーダルダイアログ）による確認メッセージ「このユーザーを無効化しますか？無効化するとログインできなくなります」を表示する
- 確認後にのみ無効化処理を実行する

### モバイル対応 🟡
**信頼性**: 🟡 *NFR-020（レスポンシブデザイン）より*
- ユーザー一覧テーブルはスマートフォン幅で横スクロールまたはカード型レイアウトに切り替える
- フォーム・ドロップダウン・ダイアログはタッチ操作に配慮したサイズ・配置とする

## 注意事項
- 本タスクで扱う管理画面はadminロール専用であるため、ルーティング・コントローラ・ビューのすべての層でアクセス制御を徹底すること（REQ-002, REQ-003と一貫した役割管理）
- 無効化（`is_active=false`）は論理的なログイン制御であり、ユーザーレコード自体は削除しない（関連データ：受注・請求書等の`created_by`/`issued_by`等の整合性を保つため）
- パスワードはbcryptでハッシュ化して保存すること（NFR-010、Laravel標準機能を利用）
- パスワードリセットフローはLaravel Breeze標準機能を最大限活用し、独自実装を避けることで保守性を高めること
- 自分自身（ログイン中の管理者自身）を無効化できないようにするなど、運用上の安全策を考慮すること（要件には明記されていないため、実装時にチームと相談の上で対応要否を判断する）
