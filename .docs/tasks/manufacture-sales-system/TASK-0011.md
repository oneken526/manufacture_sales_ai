## タスク概要
全製品の在庫状況一覧（現在庫数・引当中数・利用可能数）と、製品ごとの在庫変動履歴（日時・変動理由・数量・操作者）を閲覧する機能をwarehouse/admin向けに実装する。`InventoryController`・`StockMovementRepository`を中心に、`products`テーブルおよび`stock_movements`テーブルからデータを取得し、一覧表示・フィルタリング・在庫アラート表示を行う。本タスクは閲覧系機能であり、在庫数の更新ロジック自体は受注確定（TASK-0006）・出荷完了（TASK-0010）等の各業務フローで実装済みのものを参照する。

## 実装詳細

### 1. InventoryController実装（在庫一覧・変動履歴画面） 🔵
**信頼性**: 🔵 *api-endpoints.md（GET /inventory, GET /inventory/{product}/movements）、REQ-070、data-types.php（ProductData::availableQuantity()）より*
- `index()`: `Product`モデルを`paginate(50)`で取得し、各製品の`stock_quantity`（現在庫）・`reserved_quantity`（引当中）・`availableQuantity()`（利用可能数 = 現在庫 - 引当中）を一覧表示する。アクセス可能ロールはwarehouse, admin
- `movements(Product $product)`: 指定製品の`stock_movements`を`created_at`降順で一覧表示する。製品別・期間別（`created_at`の範囲）・理由別（`reason`）のフィルタをクエリパラメータで受け付ける
- ルーティングは`api-endpoints.md`定義の`GET /inventory`、`GET /inventory/{product}/movements`に従う
- ロールごとのアクセス制御は`EnsureUserHasRole`ミドルウェアを利用する
**実装ファイル**: `app/Http/Controllers/InventoryController.php`, `routes/web.php`

### 2. StockMovementモデル・StockMovementRepository実装、変動履歴の一覧表示・フィルタ 🟡
**信頼性**: 🟡 *REQ-072「システムは在庫数の変動履歴（日時・変動理由・数量・操作者）を記録しなければならない」、database-schema.sql（stock_movementsテーブル: product_id, reason, quantity_change, related_order_id, operated_by, created_at）、data-types.php（StockMovementReason enum: 1=引当/2=引当解除/3=出荷減算/4=返品加算/5=手動調整）から推測*
- `StockMovement`モデル: `stock_movements`テーブルに対応するEloquentモデル。`product()`（belongsTo Product）、`relatedOrder()`（belongsTo SalesOrder）、`operator()`（belongsTo User as operated_by）リレーションを定義する。`reason`属性は`StockMovementReason` enumにキャストする
- `StockMovementRepositoryInterface`（`app/Repositories/Contracts/`）: `paginateByProduct()`, `filterByReason()`, `filterByPeriod()`等のメソッドを定義
- `StockMovementRepository`（`app/Repositories/Eloquent/`）: インターフェースのEloquent実装。`idx_stock_movements_product_id`, `idx_stock_movements_created_at`インデックスを活用したクエリを構築する
- フィルタ条件（製品ID・期間・理由）はクエリビルダで組み立て、`created_at`降順でページネーション表示する
**実装ファイル**: `app/Models/StockMovement.php`, `app/Repositories/Contracts/StockMovementRepositoryInterface.php`, `app/Repositories/Eloquent/StockMovementRepository.php`

### 3. 在庫アラート（alert_threshold以下）の一覧上での強調表示 🟡
**信頼性**: 🟡 *REQ-022「在庫数が設定した閾値を下回った場合、システムは警告を表示しなければならない」、database-schema.sql（products.alert_threshold）から推測*
- `InventoryController::index()`または`Product`モデルにアラート判定ロジック（`stock_quantity <= alert_threshold`）を実装し、Bladeビューに渡す
- アラート対象製品は一覧上で行の背景色変更・警告バッジ等により強調表示する
- アラート対象件数をサマリーとして一覧上部に表示する
**実装ファイル**: `app/Http/Controllers/InventoryController.php`, `app/Models/Product.php`, `resources/views/inventory/index.blade.php`

## 単体テスト要件

### テストケース1: 在庫一覧がstock_quantity, reserved_quantity, availableQuantity()を正しく算出・表示する 🔵
**信頼性**: 🔵 *REQ-070、data-types.php（ProductData::availableQuantity() = stockQuantity - reservedQuantity）より*
- **Given**: `stock_quantity=100`, `reserved_quantity=30`の製品レコードが存在する
- **When**: `InventoryController::index()`を呼び出す
- **Then**: 一覧データに現在庫100、引当中30、利用可能数70（`availableQuantity()`の戻り値）が正しく含まれていること

### テストケース2: 変動履歴が日時降順で表示される 🟡
**信頼性**: 🟡 *REQ-072から推測*
- **Given**: 同一製品に対して異なる`created_at`の`stock_movements`レコードが複数件登録されている
- **When**: `StockMovementRepository::paginateByProduct($productId)`を呼び出す
- **Then**: 結果が`created_at`の降順（新しい順）で返却されること

### テストケース3: 在庫変動履歴が製品別・期間別・理由別に正しくフィルタされる 🟡
**信頼性**: 🟡 *REQ-072から推測されたフィルタ要件より*
- **Given**: 異なる製品・期間・理由（`reason`値）の`stock_movements`レコードが複数件登録されている
- **When**: `StockMovementRepository::filterByReason()` / `filterByPeriod()`を条件指定で呼び出す
- **Then**: 指定条件に一致するレコードのみが結果に含まれること

### テストケース4: 在庫アラート対象製品が正しく判定される 🟡
**信頼性**: 🟡 *REQ-022（alert_threshold以下で警告表示）より*
- **Given**: `stock_quantity`が`alert_threshold`以下の製品と上回る製品が混在して登録されている
- **When**: `InventoryController::index()`または`Product`のアラート判定ロジックを実行する
- **Then**: `stock_quantity <= alert_threshold`の製品のみがアラート対象として判定されること

**信頼性**: 🔵 *REQ-071「受注引き当て・出荷完了・返品・手動調整により在庫数が自動更新されなければならない」、database-schema.sql（stock_movements.reason: 1=引当/2=引当解除/3=出荷減算/4=返品加算/5=手動調整）より*

## 注意事項
- 本タスクは在庫の閲覧系機能に限定する。在庫数の更新（引当・出荷・返品・手動調整）ロジック自体はTASK-0006・TASK-0010で実装済みのものを利用し、重複実装を避けること
- `availableQuantity()`は`ProductData`（data-types.php）に定義済みのメソッドを利用し、Controller・View側での独自計算を避けて一貫性を保つこと
- 在庫変動履歴の取得は`idx_stock_movements_product_id`, `idx_stock_movements_created_at`インデックスを活用し、N+1クエリを避けるため`with('operator', 'relatedOrder')`等のEagerロードを行うこと
- ロールベースアクセス制御は`EnsureUserHasRole`ミドルウェアに委譲し、Controller内でのロール判定ロジックの重複を避けること
- アラート判定の閾値（`alert_threshold`）は製品ごとに個別設定されているため、一覧表示時は製品ごとに判定すること
