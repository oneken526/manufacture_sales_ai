## タスク概要
見積書（REQ-032）・納品書（REQ-052）・請求書（REQ-061）の3帳票で共通利用するPDF生成基盤`PdfService`を実装する。mPDFライブラリのラッパーとしてBladeテンプレートをPDFバイナリへ変換する機能、各帳票で共通のヘッダー・フッター・会社情報を含むレイアウトテンプレート、PDF生成失敗時のエラーハンドリング（EDGE-003）、および生成したPDFの保存先ディレクトリ・パス管理ロジックを実装する。本タスクで構築する基盤は、後続の見積（TASK-0008）・納品書（TASK-0010）・請求書（TASK-0012）の各PDF出力機能から利用される。

## 実装詳細

### 1. PdfServiceクラス実装（mPDFラッパー、Blade→PDF変換） 🔵
**信頼性**: 🔵 *architecture.md ディレクトリ構造「Services/PdfService.php # mPDFラッパー」、dataflow.md「PdfService(mPDF)」より*
- `PdfService`クラス（`app/Services/PdfService.php`）に`generateFromView(string $view, array $data, ?string $filename = null): string`（PDFバイナリまたは保存パスを返す）等のメソッドを実装する
- 内部で`mpdf/mpdf`をインスタンス化し、Bladeビュー（`view($view, $data)->render()`でHTML化）をmPDFに渡してPDFへ変換する
- 日本語フォント設定（mPDFのフォントディレクトリ・フォントデータ設定）を行い、日本語の帳票が正しく表示されるようにする
- `download(string $view, array $data, string $filename): Response`等、ダウンロードレスポンスを直接返すヘルパーメソッドも提供し、Controllerから簡潔に呼び出せるようにする
**実装ファイル**: `app/Services/PdfService.php`, `config/pdf.php`（mPDF関連設定がある場合）

### 2. 共通PDFレイアウト（ヘッダー・フッター・会社情報）のBladeテンプレート作成 🟡
**信頼性**: 🟡 *architecture.md ディレクトリ構造「resources/views/pdf/ # mPDF用テンプレート」より、共通レイアウトの内容は帳票要件（見積書・納品書・請求書に共通する一般的構成）から妥当に推測*
- `resources/views/pdf/layouts/base.blade.php`: 共通レイアウト（自社情報＝会社名・住所・電話番号・ロゴ等のヘッダー、ページ番号・出力日時等のフッター）を定義する
- 各帳票テンプレート（見積書・納品書・請求書、後続タスクで作成）はこの共通レイアウトを`@extends`/`@include`して利用できる構成とする
- 帳票共通のスタイル（罫線・フォントサイズ・余白等）をmPDF対応のCSSとして`resources/views/pdf/layouts/style.blade.php`（またはインラインスタイル）に定義する
- 会社情報（会社名・住所・電話番号・登録番号等）は`config/company.php`等の設定ファイルから取得できるようにし、ハードコーディングを避ける
**実装ファイル**: `resources/views/pdf/layouts/base.blade.php`, `resources/views/pdf/layouts/style.blade.php`, `config/company.php`

### 3. PDF生成失敗時のエラーハンドリング・リトライ促進メッセージ(EDGE-003) 🟡
**信頼性**: 🟡 *requirements.md EDGE-003「PDF生成に失敗した場合、システムはエラーメッセージを表示して再試行を促さなければならない」（🟡PDF機能から推測）より*
- `PdfService`内でmPDFの例外（フォント読み込み失敗・テンプレートレンダリングエラー・ファイル書き込み失敗等）を捕捉し、独自例外`PdfGenerationException`にラップしてスローする
- 例外には「PDFの生成に失敗しました。しばらく経ってから再度お試しください」等、ユーザー向けの再試行を促すメッセージを含める
- Controller層で`PdfGenerationException`を捕捉し、画面にフラッシュメッセージとして表示する仕組みの基盤（共通の例外ハンドラまたはトレイト）を用意する
- 失敗時はログ（`Log::error()`）に詳細なエラー内容（テンプレート名・例外メッセージ・スタックトレース）を記録し、運用調査を可能にする
**実装ファイル**: `app/Services/PdfService.php`, `app/Exceptions/PdfGenerationException.php`, `app/Exceptions/Handler.php`（または例外レンダリングロジック）

### 4. PDF保存先ディレクトリ・パス管理ロジック(storage配下) 🟡
**信頼性**: 🟡 *architecture.md「PDF生成（帳票枚数が多い場合）は必要に応じてキューで非同期化を検討する」、一般的なLaravelファイル管理慣習から推測*
- PDFファイルの保存先を`storage/app/pdf/{帳票種別}/{年度}/`のような階層構造で管理し、`Storage`ファサード（`local`または`public`ディスク）を用いて保存する
- ファイル名の命名規則（例: 見積書`quotation_{採番番号}.pdf`、納品書`shipment_{出荷ID}.pdf`、請求書`invoice_{請求書番号}.pdf`）を`PdfService`内のヘルパーメソッドとして一元管理し、後続タスクから一貫した方法でパスを取得できるようにする
- ダウンロード時は保存済みファイルを再利用するか、都度生成するかの方針（本タスクでは都度生成を基本とし、保存は履歴・再ダウンロード用途とする）を整理しコメントで明記する
**実装ファイル**: `app/Services/PdfService.php`, `config/filesystems.php`

## 注意事項
- mPDFの日本語フォント設定（フォントファイルの配置・`fontDir`/`fontdata`設定）を誤ると文字化けが発生するため、必ず日本語を含むテンプレートでテストを行うこと
- `PdfService`は帳票種別（見積書・納品書・請求書）に依存しない汎用的なインターフェースとして設計し、帳票固有のレイアウト・データ整形は各帳票テンプレート側（後続タスク）に委ねること（責務分離）
- PDF生成は処理負荷が高い操作であるため、将来的なキュー非同期化（architecture.md記載）を考慮し、`PdfService`の呼び出し方法（同期/非同期）を切り替えやすい設計にしておくこと
- 生成したPDFを`storage/app/pdf/`に保存する際は、ディスク容量の増大に備えて命名規則・ディレクトリ構成を明確にし、将来的なクリーンアップ（古いファイルの削除）を考慮した設計とすること
- 例外メッセージ・ログ出力には個人情報（顧客情報等）を含めないよう注意すること
