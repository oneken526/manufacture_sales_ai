## タスク概要
受注の一覧表示（ステータスフィルタ）・詳細表示・編集・キャンセル・出荷指示発行を実装する。受注確定後の編集はシステム管理者（admin）のみに限定し（REQ-042）、キャンセル時には在庫引当を正しく解除（`reserved_quantity`の減算・`stock_movements`記録・ステータス更新）するロジックをDBトランザクションでアトミックに実行する（REQ-043）。

## 実装詳細

### 1. SalesOrder/SalesOrderItemモデル・SalesOrderRepository・OrderService実装 🔵
**信頼性**: 🔵 *architecture.mdのRepository+Serviceパターン、database-schema.sql（sales_orders, sales_order_itemsテーブル定義: status CHECK制約 BETWEEN 1 AND 6）より*
- `SalesOrder`モデル（`app/Models/SalesOrder.php`）: `customer`（belongsTo Customer）, `quotation`（belongsTo Quotation）, `items`（hasMany SalesOrderItem）, `createdBy`（belongsTo User）等のリレーションと、ステータスコード（1=confirmed, 2=shipping_instructed, 3=shipped, 4=invoiced, 5=cancelled, 6=returned）に対応する定数・スコープ（`scopeStatus()`等）を定義する
- `SalesOrderItem`モデル（`app/Models/SalesOrderItem.php`）: `order`（belongsTo SalesOrder）, `product`（belongsTo Product）のリレーションを定義する
- `SalesOrderRepositoryInterface`と`EloquentSalesOrderRepository`（`app/Repositories/`配下）を実装し、ステータスフィルタ付き一覧取得（ページネーション50件 NFR-021）・詳細取得・更新・キャンセル時のロック付き取得を担う
- `OrderService`（`app/Services/OrderService.php`）: 受注詳細取得・編集・キャンセル（`cancel()`）・出荷指示発行（`issueShippingInstruction()`）のユースケースを集約する
**実装ファイル**: `app/Models/SalesOrder.php`, `app/Models/SalesOrderItem.php`, `app/Repositories/SalesOrderRepositoryInterface.php`, `app/Repositories/EloquentSalesOrderRepository.php`, `app/Services/OrderService.php`

### 2. OrderController実装（一覧[ステータスフィルタ]・詳細・編集・キャンセル・出荷指示発行） 🔵
**信頼性**: 🔵 *api-endpoints.md「受注管理」セクション（GET /orders, GET /orders/{order}, PUT /orders/{order}, POST /orders/{order}/cancel, POST /orders/{order}/shipping-instruction）より*
- `index()`: 受注一覧表示。クエリパラメータ`status`によるステータスフィルタ（confirmed/shipping_instructed/shipped/invoiced/cancelled/returned）と顧客名検索、ページネーション（50件）を実装する
- `show()`: 受注詳細表示（明細・顧客情報・ステータス履歴・関連見積へのリンクを表示）
- `edit()` / `update()`: 受注編集フォーム表示・更新処理（確定後はadminのみ許可、`OrderPolicy`で制御）
- `cancel()`: キャンセル処理（`POST /orders/{order}/cancel`、`OrderService::cancel()`を呼び出す。キャンセル理由の入力を任意で受け付ける）
- `issueShippingInstruction()`: 出荷指示発行処理（`POST /orders/{order}/shipping-instruction`、ステータスを`confirmed`→`shipping_instructed`に変更）
- ルーティングは一覧・詳細を`middleware(['auth', 'role:sales,accounting,admin'])`、編集を`role:admin`、キャンセル・出荷指示発行を`role:sales,admin`に限定する（api-endpoints.mdの権限定義と整合）
**実装ファイル**: `app/Http/Controllers/OrderController.php`, `app/Http/Requests/UpdateSalesOrderRequest.php`, `routes/web.php`

### 3. 受注編集制限: 確定後はシステム管理者(admin)のみ編集可能、Policyで制御 🟡
**信頼性**: 🟡 *REQ-042「受注確定後はシステム管理者のみが内容編集できなければならない」（要件定義書では🟡 業務フロー管理から推測と明記）、api-endpoints.md（PUT /orders/{order}: 権限=admin）より*
- `OrderPolicy`（`app/Policies/OrderPolicy.php`）を実装し、`update()`メソッドで「ログインユーザーが`admin`ロールであること」を判定する
- `OrderController::edit()`/`update()`では`$this->authorize('update', $order)`によりPolicyを適用し、admin以外のユーザーがアクセスした場合は403を返す
- 確定前（draftに相当する状態が存在する場合、または見積段階）の編集制限がある場合は別途要件を確認する。本タスクでは「受注として確定済み（`status >= 1`）の受注」を編集制限の対象とする
- Bladeビューでは、ログインユーザーがadminロールでない場合に編集ボタン自体を非表示にする（UI制御と認可制御の二重化）
**実装ファイル**: `app/Policies/OrderPolicy.php`, `app/Providers/AuthServiceProvider.php`, `app/Http/Controllers/OrderController.php`

### 4. キャンセル処理: 引当解除(products.reserved_quantity減算)、stock_movements記録(reason=release=2)、ステータスをcancelled(=5)に変更、cancelled_at記録をDBトランザクションで実行 🔵
**信頼性**: 🔵 *REQ-043「システムはキャンセル処理ができ、キャンセル時は引き当て在庫を戻さなければならない」、database-schema.sql（sales_orders.status: 5=キャンセル(cancelled)、stock_movements.reason: 2=引き当て解除(release)）、dataflow.md「受注ステータス遷移」（受注確定→キャンセル：引当解除 REQ-043）より*
- `OrderService::cancel(SalesOrder $order, ?string $reason = null)`を実装し、以下を単一のDBトランザクション（`DB::transaction()`）内でアトミックに実行する:
  1. キャンセル可能なステータス（`confirmed`(=1)または`shipping_instructed`(=2)）であることを確認する。出荷完了後（`shipped`以降）のキャンセルは不可とし、業務エラーを返す
  2. 受注明細に紐づく各製品を`lockForUpdate()`で取得し、`reserved_quantity`を明細数量分だけ減算する（UPDATE products）
  3. `stock_movements`に在庫変動履歴を記録する（`reason=2`(release)、`quantity`・`operated_by`・`related_order_id`等を記録）
  4. `sales_orders.status`を`5`（cancelled）に更新し、`cancelled_at`に現在時刻を記録する（キャンセル理由を保持するカラムがある場合は`cancel_reason`等にも記録する）
- 減算後に`reserved_quantity`が負値にならないことをアプリケーションロジックで検証する（万一の不整合時は例外をスローしロールバックする）
- トランザクション失敗時は全体をロールバックする
**実装ファイル**: `app/Services/OrderService.php`, `app/Repositories/EloquentProductRepository.php`, `app/Models/StockMovement.php`, `app/Http/Controllers/OrderController.php`

### 5. 出荷指示発行処理: ステータスをconfirmed(1)→shipping_instructed(2)へ変更 🔵
**信頼性**: 🔵 *REQ-041「システムは受注から出荷指示を発行できなければならない」、database-schema.sql（sales_orders.status: 1=受注確定(confirmed)/2=出荷指示済み(shipping_instructed)）、dataflow.md「受注ステータス遷移」（受注確定→出荷指示済み: 出荷指示発行 REQ-041）より*
- `OrderService::issueShippingInstruction(SalesOrder $order)`を実装し、対象受注のステータスが`confirmed`(=1)であることを確認したうえで`shipping_instructed`(=2)へ更新する
- ステータス遷移として不正な状態（すでに出荷指示済み・出荷完了済み・キャンセル済み等）からの発行操作はエラーとして扱い、適切なメッセージを返す
- 発行後、対象受注はTASK-0010の出荷指示一覧（`GET /shipments`）に表示されるようになる
**実装ファイル**: `app/Services/OrderService.php`, `app/Http/Controllers/OrderController.php`

## 単体テスト要件

### テストケース1: OrderService::cancel()が引当解除と在庫整合性(reserved_quantity>=0)を保つ 🔵
**信頼性**: 🔵 *REQ-043、dataflow.md「データ整合性の保証」（利用可能在庫の整合性チェック・マイナスにならないことを保証）より*
- **Given**: `confirmed`(=1)ステータスの受注が存在し、関連製品の`reserved_quantity`が当該受注の明細数量を含む値である
- **When**: `OrderService::cancel($order)`を実行する
- **Then**: 各製品の`reserved_quantity`が明細数量分だけ正しく減算され、結果が`0`以上であること。`stock_movements`に`reason=2`(release)のレコードが記録され、`sales_orders.status`が`5`(cancelled)に変更され、`cancelled_at`に現在時刻が記録されること。これらすべてが単一のDBトランザクション内で実行され、一部でも失敗した場合は全件ロールバックされること

### テストケース2: 出荷完了後の受注はキャンセルできない 🔵
**信頼性**: 🔵 *dataflow.md「受注ステータス遷移」（出荷完了→請求済み/返品済みへの遷移は定義されているが、出荷完了後のキャンセルへの遷移は定義されていない）、REQ-043の趣旨（在庫引当の解除を前提としたキャンセル）より*
- **Given**: `shipped`(=3)以降のステータス（出荷完了・請求済み・返品済み等）にある受注が存在する
- **When**: `OrderService::cancel($order)`を実行する
- **Then**: キャンセル不可を示す業務エラー（例外）が返却され、ステータス・在庫情報・`stock_movements`のいずれにも変更が加わらないこと

### テストケース3: admin以外が確定済み受注を編集しようとすると拒否される 🟡
**信頼性**: 🟡 *REQ-042（要件定義書では🟡 業務フロー管理から推測と明記）、api-endpoints.md（PUT /orders/{order}: 権限=admin）より*
- **Given**: sales, warehouse, accountingいずれかのロールを持つユーザーがログインしており、`confirmed`(=1)以降のステータスにある受注が存在する
- **When**: 当該ユーザーが受注編集画面（`PUT /orders/{order}`）への更新リクエストを送信する
- **Then**: `OrderPolicy::update()`により認可が拒否され、403エラーが返却され、受注内容が更新されないこと

### テストケース4: 出荷指示発行によりステータスがconfirmedからshipping_instructedへ正しく遷移する 🔵
**信頼性**: 🔵 *REQ-041、database-schema.sql（sales_orders.status コード表）、dataflow.md「受注ステータス遷移」より*
- **Given**: `confirmed`(=1)ステータスの受注が存在する
- **When**: `OrderService::issueShippingInstruction($order)`を実行する
- **Then**: `sales_orders.status`が`2`（shipping_instructed）に更新されること。すでに`shipping_instructed`以降のステータスにある受注に対して再実行した場合はエラーとなり、ステータスが変更されないこと

## 注意事項
- キャンセル処理（在庫引当解除を伴う）は必ずDBトランザクション（`DB::transaction()`）内で実行し、対象製品レコードには`lockForUpdate()`による悲観的ロックを適用すること（dataflow.md「データ整合性の保証」と整合）
- キャンセル可能なステータスの範囲（出荷完了後のキャンセルを許可するか等）は要件定義書に明記がないため、実装時にチームと確認すること。本タスクでは出荷完了前（`confirmed`/`shipping_instructed`）のみキャンセル可能とする方針で設計している
- 受注編集制限（REQ-042）は要件定義書でも🟡（推測）扱いであり、「確定後」の定義（受注確定後すべての状態を含むか、特定ステータスのみか）について実装時に確認することが望ましい
- TASK-0008（見積管理機能）の`QuotationService::confirmToOrder()`との責務境界（受注作成は見積側、その後の編集・キャンセル・出荷指示は本タスクの`OrderService`）を明確にし、重複実装を避けること
- ステータス遷移はdataflow.md「受注ステータス遷移」のステートマシン図に厳密に従い、不正な遷移（例: キャンセル済みから出荷指示発行）を防止するガード処理を各サービスメソッドに実装すること
