## タスク概要
出荷完了後の受注に対する請求書発行・PDF出力・入金ステータス管理機能を実装する（accounting/admin向け、REQ-064で権限制御）。`Invoice`モデル・`InvoiceRepository`・`InvoiceService`を中心に、年度+連番方式の請求書番号採番（`document_sequences`活用）、二重請求発行防止（EDGE-004）、`PdfService`（TASK-0004）を活用した請求書PDF生成、入金ステータス（未入金/部分入金/入金済み）の手動更新機能までを縦に貫通して実装する。dataflow.md「機能3: 請求書発行→PDF出力」のシーケンスに従い、`InvoiceController → InvoiceService → Repository/PdfService → DB`の流れで処理を行う。

## 実装詳細

### 1. Invoiceモデル・InvoiceRepository・InvoiceService実装 🔵
**信頼性**: 🔵 *architecture.md（Invoice.php / InvoiceService.php構成、Repository+Serviceパターン）、database-schema.sql（invoicesテーブル定義）、data-types.php（InvoiceData DTO, PaymentStatus enum）より*
- `Invoice`モデル: `invoices`テーブルに対応するEloquentモデル。`invoice_number`, `sales_order_id`, `total_amount`, `payment_status`, `invoice_pdf_path`, `issued_at`, `issued_by`を`$fillable`に設定し、`payment_status`属性は`PaymentStatus` enum（1=未入金/2=部分入金/3=入金済み）にキャストする。`salesOrder()`（belongsTo）、`payments()`（hasMany Payment）、`issuedBy()`（belongsTo User）リレーションを定義する
- `InvoiceRepositoryInterface`（`app/Repositories/Contracts/`）: `paginateByPaymentStatus()`, `findBySalesOrderId()`, `create()`, `updatePaymentStatus()`等のメソッドを定義
- `InvoiceRepository`（`app/Repositories/Eloquent/`）: インターフェースのEloquent実装。`idx_invoices_payment_status`インデックスを活用したクエリを構築する
- `InvoiceService`: 請求書発行（`issue()`）・採番・PDF生成依頼・入金ステータス更新の業務ロジックを集約する。`InvoiceData` DTO（既にdata-types.phpに定義済み）を用いてController⇔Service間のデータ受け渡しを行う
**実装ファイル**: `app/Models/Invoice.php`, `app/Repositories/Contracts/InvoiceRepositoryInterface.php`, `app/Repositories/Eloquent/InvoiceRepository.php`, `app/Services/InvoiceService.php`

### 2. InvoiceController実装（一覧[入金ステータスフィルタ]・発行・PDF・入金ステータス手動更新） 🔵
**信頼性**: 🔵 *api-endpoints.md（GET /invoices, POST /invoices/{order}, GET /invoices/{invoice}/pdf, PUT /invoices/{invoice}/payment-status）、REQ-060〜062、dataflow.md「機能3」より*
- `index()`: `GET /invoices` - 請求書一覧を`payment_status`によるフィルタ付きで`paginate(50)`表示する。アクセス可能ロールはaccounting, admin
- `store(SalesOrder $order)`: `POST /invoices/{order}` - `InvoiceService::issue($order)`を呼び出し請求書を発行する。出荷完了済み（`status=3`）でない受注、または既発行の受注に対しては適切なエラーを返す。アクセス可能ロールはaccounting, admin
- `pdf(Invoice $invoice)`: `GET /invoices/{invoice}/pdf` - 生成済み請求書PDFをダウンロードレスポンスとして返す。アクセス可能ロールはaccounting, sales, admin
- `updatePaymentStatus(Invoice $invoice, UpdatePaymentStatusRequest $request)`: `PUT /invoices/{invoice}/payment-status` - 入金ステータス（未入金/部分入金/入金済み）を手動更新する。アクセス可能ロールはaccounting, admin
- ロールごとのアクセス制御は`EnsureUserHasRole`ミドルウェアを利用する
**実装ファイル**: `app/Http/Controllers/InvoiceController.php`, `app/Http/Requests/UpdatePaymentStatusRequest.php`, `routes/web.php`

### 3. 請求書番号採番ロジック（document_sequencesテーブル活用、年度+連番、例: INV-2026-0001） 🔵
**信頼性**: 🔵 *設計ヒアリングQ5、database-schema.sql（document_sequencesテーブル: document_type=3が請求書、UNIQUE(document_type, fiscal_year)）、data-types.php（DocumentType::INVOICE = 3）より*
- `InvoiceService`内（または共通の`DocumentNumberGenerator`サービス）に採番ロジックを実装する: `document_sequences`テーブルから`document_type=3（INVOICE）`・対象年度のレコードを取得し、`last_number`をインクリメントしたうえで`INV-{年度}-{連番4桁ゼロ埋め}`形式（例: INV-2026-0001）の文字列を生成する
- 採番処理はトランザクション内で行レベルロック（`lockForUpdate()`）を用いて競合を防止する（同時発行時の番号重複防止）
- 年度をまたいだ場合は`document_sequences`に新規の`(document_type=3, fiscal_year)`レコードを作成し、連番を1からリセットする
**実装ファイル**: `app/Services/InvoiceService.php`, `app/Services/DocumentNumberGenerator.php`, `app/Models/DocumentSequence.php`

### 4. 二重請求発行防止: invoices.sales_order_id UNIQUE制約 + アプリ側の事前チェック（EDGE-004） 🔵
**信頼性**: 🔵 *EDGE-004「同一受注に対して二重請求発行を防止しなければならない」、database-schema.sql（invoices.sales_order_id BIGINT NOT NULL UNIQUE）、dataflow.md「機能3」（既存請求書の有無を確認→既発行の場合は重複エラー）より*
- `InvoiceService::issue(SalesOrder $order)`: 発行前に`InvoiceRepository::findBySalesOrderId($order->id)`で既存請求書の有無を確認する（アプリ側事前チェック）
- 既に発行済みの場合は`DuplicateInvoiceException`をスローし、コントローラ側で警告メッセージ「この受注の請求書は既に発行済みです」を表示する（dataflow.md「機能3」のalt分岐）
- データベースレベルでも`invoices.sales_order_id`のUNIQUE制約により二重INSERTを防止する（多重リクエスト等の競合状態への最終防衛線）
- アプリ側チェックとDB制約の両方による二重防止により、表示上のエラーハンドリングとデータ整合性を両立する
**実装ファイル**: `app/Services/InvoiceService.php`, `app/Exceptions/DuplicateInvoiceException.php`

### 5. 請求書PDF生成（PdfService(TASK-0004)を活用したInvoicePdfテンプレート） 🔵
**信頼性**: 🔵 *REQ-061「請求書はPDF形式でダウンロードできなければならない」、architecture.md（PdfService.php: mPDFラッパー）、dataflow.md「機能3」（S->>P: 請求書PDF生成を依頼 / P-->>S: PDFファイル）より*
- `InvoiceService::issue()`内で`PdfService::generate('invoices.pdf', $data)`（TASK-0004で実装済みのラッパー）を呼び出し、請求書PDFを生成する
- `resources/views/invoices/pdf.blade.php`（InvoicePdfテンプレート）: 請求書番号・宛先（顧客情報）・請求金額・明細（受注明細）・発行日等を含む請求書帳票レイアウトをBladeで定義する
- 生成されたPDFは`storage/app/invoices/`配下等に保存し、そのパスを`invoices.invoice_pdf_path`に記録する
- ダウンロードは`InvoiceController::pdf()`から保存済みPDFファイルをレスポンスとして返す
**実装ファイル**: `app/Services/InvoiceService.php`, `resources/views/invoices/pdf.blade.php`

### 6. accounting/admin以外のアクセス制限（REQ-064） 🔵
**信頼性**: 🔵 *REQ-064「管理職・経理担当者のみが請求書を発行・入金確認できなければならない」、api-endpoints.mdのロール定義（accounting, admin / PDFダウンロードのみsalesも含む）より*
- `InvoiceController`の発行（`store`）・入金ステータス更新（`updatePaymentStatus`）・一覧（`index`）アクションには`EnsureUserHasRole:accounting,admin`ミドルウェアを適用する
- PDFダウンロード（`pdf`）のみsalesロールにも閲覧を許可する（api-endpoints.md定義に従う）
- 権限のないロールがアクセスした場合は403エラー（権限エラー画面）を返す
**実装ファイル**: `routes/web.php`, `app/Http/Controllers/InvoiceController.php`

### 7. 入金ステータス（unpaid/partial/paid）の手動更新機能 🔵
**信頼性**: 🔵 *REQ-062「システムは入金ステータス（未入金・部分入金・入金済み）を管理できなければならない」、data-types.php（PaymentStatus enum: UNPAID=1/PARTIALLY_PAID=2/PAID=3）より*
- `InvoiceService::updatePaymentStatus(Invoice $invoice, PaymentStatus $status)`: `payment_status`カラムを更新する
- 更新フォームでは未入金/部分入金/入金済みの3択から選択できるUIを提供する
- 更新履歴を残す必要がある場合は`payments`テーブルへの手動入金記録（`source=1: manual`）と連携する設計とする
**実装ファイル**: `app/Services/InvoiceService.php`, `app/Http/Controllers/InvoiceController.php`, `resources/views/invoices/show.blade.php`

## 単体テスト要件

### テストケース1: InvoiceService::issue()が同一受注への二重発行を防止する（EDGE-004） 🔵
**信頼性**: 🔵 *EDGE-004、database-schema.sql（invoices.sales_order_id UNIQUE）より*
- **Given**: ある受注（`sales_order_id=1`）に対して既に請求書が発行済みである
- **When**: 同じ受注に対して`InvoiceService::issue($order)`を再度呼び出す
- **Then**: `DuplicateInvoiceException`がスローされ、新たな`invoices`レコードが作成されないこと

### テストケース2: 採番が年度をまたぐとリセットされる 🔵
**信頼性**: 🔵 *設計ヒアリングQ5、database-schema.sql（document_sequences: UNIQUE(document_type, fiscal_year)、last_number）より*
- **Given**: `document_sequences`に`(document_type=3, fiscal_year=2025, last_number=120)`のレコードが存在し、現在の年度が2026年である
- **When**: 2026年度の請求書発行で`InvoiceService`の採番ロジックを実行する
- **Then**: `(document_type=3, fiscal_year=2026)`の新規レコードが作成され、`last_number=1`から採番が開始され、請求書番号が`INV-2026-0001`となること

### テストケース3: accounting以外がissueを呼ぶと権限エラーになる（REQ-064） 🔵
**信頼性**: 🔵 *REQ-064「管理職・経理担当者のみが請求書を発行・入金確認できなければならない」より*
- **Given**: salesロールまたはwarehouseロールのユーザーでログインしている
- **When**: `POST /invoices/{order}`（請求書発行）または`PUT /invoices/{invoice}/payment-status`（入金ステータス更新）を実行する
- **Then**: 403エラー（権限エラー）が返却され、請求書の発行・更新が行われないこと

### テストケース4: 出荷未完了の受注に対する請求書発行が拒否される 🟡
**信頼性**: 🟡 *REQ-060「出荷完了後、システムは請求書を発行できなければならない」から推測*
- **Given**: ステータスが「出荷完了」（`status=3`）でない受注（例: 受注確定のみ）が存在する
- **When**: `InvoiceService::issue($order)`を呼び出す
- **Then**: 適切な例外がスローされ、請求書が発行されないこと

## 注意事項
- 二重発行防止はアプリ側チェックとDB UNIQUE制約の両方を必ず実装すること（EDGE-004）。アプリ側チェックのみでは競合状態（同時リクエスト）での重複を防げないため、最終防衛線としてDB制約が必須である
- 採番処理は`document_sequences`テーブルへの行ロック（`lockForUpdate()`）とトランザクションを用いて、同時発行時の番号重複を防止すること
- `PdfService`（TASK-0004）は既存実装を再利用し、PDFテンプレート（Bladeビュー）のみを新規作成すること。mPDFラッパーの重複実装を避けること
- ロールベースアクセス制御は`EnsureUserHasRole`ミドルウェアに委譲し、Controller内でのロール判定ロジックの重複を避けること（REQ-064）
- 出荷完了（`shipments.shipped_at`が設定済み、`sales_orders.status=3`）の受注のみを請求書発行対象とすること（REQ-060の前提条件）
