## タスク概要
顧客（取引先）マスタのCRUD機能一式（一覧・登録・編集・詳細・削除・検索）を、DB（customersテーブル）からモデル・リポジトリ・サービス・コントローラ・Blade画面までを縦に貫通して実装する。Repository+Serviceパターンに従い、業務ロジックはCustomerServiceに集約し、データアクセスはCustomerRepositoryに抽象化する。検索機能・受注存在チェックによる削除制限・受注履歴表示など、顧客マスタを起点とした営業業務フローの土台を構築する。

## 実装詳細

### 1. Customerモデル・CustomerRepository・CustomerService実装 🔵
**信頼性**: 🔵 *architecture.md（レイヤードアーキテクチャ + リポジトリパターン: Controller → Service → Repository → Eloquent Model → DB）、database-schema.sql（customersテーブル定義）より*
- `Customer`モデル: `customers`テーブルに対応するEloquentモデル。`SoftDeletes`トレイトを使用し`deleted_at`を扱う。`company_name`, `contact_name`, `address`, `phone`, `email`, `credit_limit`を`$fillable`に設定。`salesOrders()`リレーション（hasMany）を定義する
- `CustomerRepositoryInterface`（`app/Repositories/Contracts/`）: `paginate()`, `find()`, `create()`, `update()`, `delete()`, `search()`, `hasOrders()`等のメソッドを定義
- `CustomerRepository`（`app/Repositories/Eloquent/`）: インターフェースのEloquent実装。データアクセスを抽象化しテスト時にモック化しやすくする
- `CustomerService`: 業務ロジックを集約。CRUD操作・検索・削除可否判定（受注存在チェック）を提供する
- `CustomerData` DTO（既にarchitecture.mdに定義済み）を用いてController⇔Service間のデータ受け渡しを行う
**実装ファイル**: `app/Models/Customer.php`, `app/Repositories/Contracts/CustomerRepositoryInterface.php`, `app/Repositories/Eloquent/CustomerRepository.php`, `app/Services/CustomerService.php`

### 2. CustomerController実装（一覧・登録・編集・詳細・削除） 🔵
**信頼性**: 🔵 *api-endpoints.md（GET /customers, GET /customers/create, POST /customers, GET /customers/{customer}, GET /customers/{customer}/edit, PUT /customers/{customer}, DELETE /customers/{customer}）、NFR-021（ページネーション1ページ50件）より*
- `index()`: 顧客一覧を取得し、`paginate(50)`でページネーション表示する。検索キーワードがある場合は検索結果を表示する。アクセス可能ロールは sales, accounting, admin
- `create()` / `store()`: 新規登録フォーム表示・登録処理（バリデーション含む）。アクセス可能ロールは sales, admin
- `show()`: 顧客詳細表示（受注履歴一覧を含む）。アクセス可能ロールは sales, accounting, admin
- `edit()` / `update()`: 編集フォーム表示・更新処理。アクセス可能ロールは sales, admin
- `destroy()`: 削除処理（受注存在時は拒否しエラーを返す）。アクセス可能ロールは admin のみ
- ロールごとのアクセス制御は`EnsureUserHasRole`ミドルウェアを利用する
**実装ファイル**: `app/Http/Controllers/CustomerController.php`, `app/Http/Requests/StoreCustomerRequest.php`, `app/Http/Requests/UpdateCustomerRequest.php`, `routes/web.php`

### 3. 顧客検索機能（会社名・担当者名・電話番号での部分一致検索） 🟡
**信頼性**: 🟡 *REQ-011「システムは顧客を検索（会社名・担当者名・電話番号）できなければならない」より、検索方式（部分一致）はマスタ管理の一般的UXから妥当に推測*
- `CustomerRepository::search(string $keyword)`: `company_name`, `contact_name`, `phone`に対して`LIKE '%keyword%'`によるOR検索を行う
- 検索キーワードはcrequest queryパラメータ（`q`）として受け取り、`index()`画面のフィルタとして適用する
- 内部AJAX用エンドポイント `GET /api/internal/customers/search?q={keyword}`（見積作成画面のオートコンプリート用、api-endpoints.mdに定義）と連携可能な設計とする
- 検索条件は`idx_customers_company_name`インデックス（database-schema.sql）を活用しパフォーマンスに配慮する
**実装ファイル**: `app/Repositories/Eloquent/CustomerRepository.php`, `app/Http/Controllers/CustomerController.php`

### 4. 受注存在チェックによる削除制限・ソフトデリート 🟡
**信頼性**: 🟡 *REQ-012「受注が存在する顧客の場合、システムは削除を禁止し警告を表示しなければならない」、database-schema.sql（customers.deleted_at: 🟡 受注がある顧客は削除禁止のためソフトデリート推奨）より*
- `CustomerService::delete(int $customerId)`: 削除前に`CustomerRepository::hasOrders($customerId)`（`sales_orders`テーブルに当該`customer_id`のレコードが存在するか）をチェックする
- 受注が存在する場合は専用の例外（例: `CustomerHasOrdersException`）をスローし、コントローラ側でエラーメッセージ「この顧客には受注履歴があるため削除できません」を表示する
- 受注が存在しない場合のみ`Customer`モデルの`SoftDeletes`機能により論理削除（`deleted_at`にタイムスタンプを記録）を実行する
- 物理削除は行わず、データの履歴・整合性を保持する
**実装ファイル**: `app/Services/CustomerService.php`, `app/Exceptions/CustomerHasOrdersException.php`

### 5. 顧客詳細画面に受注履歴一覧表示 🟡
**信頼性**: 🟡 *REQ-013「システムは顧客ごとの受注履歴を一覧表示できなければならない」、api-endpoints.md（GET /customers/{customer} 顧客詳細（受注履歴含む））より、営業担当の業務フローから妥当に推測*
- `CustomerController::show()`: `Customer`モデルの`salesOrders`リレーションをEagerロード（`with('salesOrders')`）し、受注番号・受注日・ステータス・合計金額等を一覧表示する
- 受注履歴は新しい順にソートし、件数が多い場合はページネーションまたは直近N件表示とする
- 各受注行から受注詳細画面（OrderController管轄、別タスク）へのリンクを設置する
**実装ファイル**: `app/Http/Controllers/CustomerController.php`, `resources/views/customers/show.blade.php`

### 6. Blade画面（一覧/登録フォーム/編集フォーム/詳細）+ jQueryインクリメンタルサーチ 🔵
**信頼性**: 🔵 *architecture.md（resources/views/customers/, resources/js/ jQueryスクリプト群構成）、api-endpoints.md（顧客管理画面エンドポイント一覧）より*
- `index.blade.php`: 顧客一覧テーブル + 検索フォーム + ページネーションリンク
- `create.blade.php` / `edit.blade.php`: 顧客登録・編集フォーム（会社名・担当者名・住所・電話・メール・与信枠の入力欄、バリデーションエラー表示）
- `show.blade.php`: 顧客詳細情報 + 受注履歴一覧
- jQueryによる検索インクリメンタルサーチ: 検索フォームの`keyup`イベントでAJAXリクエスト（`/api/internal/customers/search`）を送信し、結果を非同期に一覧へ反映する。デバウンス処理を入れ過剰なリクエストを防止する
**実装ファイル**: `resources/views/customers/index.blade.php`, `resources/views/customers/create.blade.php`, `resources/views/customers/edit.blade.php`, `resources/views/customers/show.blade.php`, `resources/js/customers-search.js`

## 単体テスト要件

### テストケース1: CustomerServiceのCRUD操作が正しく動作する 🔵
**信頼性**: 🔵 *REQ-010（顧客情報の登録・編集・削除）より*
- **Given**: 有効な顧客データ（会社名・担当者名・住所・電話・メール・与信枠）が用意されている
- **When**: `CustomerService::create()` / `update()` / `find()` / `delete()`を呼び出す
- **Then**: それぞれの操作がデータベースに正しく反映され、取得結果が入力データと一致すること

### テストケース2: 受注が存在する顧客の削除が拒否される 🟡
**信頼性**: 🟡 *REQ-012より*
- **Given**: `sales_orders`テーブルに紐づく受注レコードが存在する顧客がいる
- **When**: `CustomerService::delete($customerId)`を呼び出す
- **Then**: `CustomerHasOrdersException`がスローされ、顧客レコードは削除されず、`deleted_at`もNULLのままであること

### テストケース3: 検索機能が会社名・担当者名・電話番号で部分一致検索できる 🟡
**信頼性**: 🟡 *REQ-011より*
- **Given**: 複数の顧客レコード（会社名・担当者名・電話番号が異なる）が登録されている
- **When**: `CustomerRepository::search('キーワード')`を呼び出す
- **Then**: 会社名・担当者名・電話番号のいずれかにキーワードを部分一致で含む顧客のみが結果に含まれること

### テストケース4: 一覧取得が50件単位でページネーションされる 🔵
**信頼性**: 🔵 *NFR-021（ページネーション1ページ50件）より*
- **Given**: 顧客レコードが60件登録されている
- **When**: `CustomerService::paginate()`（または`CustomerController::index()`）を呼び出す
- **Then**: 1ページ目に50件、2ページ目に10件が返却されること

## 注意事項
- 削除は必ずソフトデリート（`deleted_at`）で行い、物理削除は行わないこと（受注等の関連データとの整合性維持のため）
- 受注存在チェックは`sales_orders.customer_id`に対する存在確認で行い、N+1クエリを避けるためインデックス（`idx_sales_orders_customer_id`）を活用すること
- 検索処理はSQLインジェクション対策としてEloquent ORMのクエリビルダを使用すること（NFR-013）
- ロールベースアクセス制御は`EnsureUserHasRole`ミドルウェアに委譲し、Controller内でのロール判定ロジックの重複を避けること
- 与信枠（`credit_limit`）はBIGINT型で管理されるため、入力値のバリデーション（数値・桁数）に注意すること
