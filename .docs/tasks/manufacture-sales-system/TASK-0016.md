## 実装詳細

### 1. 主要業務フロー（見積作成→受注確定→出荷完了→請求発行→入金確認）のE2E/Featureテスト実装 🔵
**信頼性**: 🔵 *acceptance-criteria.md（業務フロー全体を通した受け入れ基準）、dataflow.md（見積→受注→出荷→請求→入金の一連シーケンス）より*
- Laravelの`Feature`テスト（`tests/Feature/`）として、見積作成（TASK-0008）→見積からの受注確定（TASK-0009、在庫引当）→出荷指示・出荷完了（TASK-0010, TASK-0011、在庫減算）→請求書発行（TASK-0012）→入金CSV取込・消込確認（TASK-0013）までを通しで検証するテストクラスを実装する
- 各ステップでのステータス遷移（見積: draft→sent→accepted等、受注: pending→confirmed→shipped→completed等、請求書: unissued→issued→paid等）が正しく行われることを検証する
- 各ステップ完了後にデータベース上のレコード状態（在庫数、引当数、請求金額、入金消込状態等）が期待通りであることを`assertDatabaseHas`等でアサートする
**実装ファイル**: `tests/Feature/SalesFlowEndToEndTest.php`, `tests/Feature/QuotationToOrderFlowTest.php`, `tests/Feature/ShippingToInvoiceFlowTest.php`

### 2. 4役割×主要機能のアクセス制御統合テスト実装 🔵
**信頼性**: 🔵 *REQ-002「システムは4つの役割（システム管理者・営業担当者・在庫出荷担当者・管理職経理担当）を持たなければならない」、REQ-003「在庫・出荷担当者が請求書操作を実行しようとした場合、システムはアクセスを拒否しなければならない」、REQ-064「管理職・経理担当者のみが請求書を発行・入金確認できなければならない」、api-endpoints.md「権限制御」より*
- `admin`, `sales`, `warehouse`, `accounting`の4役割それぞれについて、顧客・製品・見積・受注・出荷・請求書・レポート・内部APIの各機能へのアクセス可否をマトリクス形式で網羅的にテストする
- `warehouse`ロールが請求書操作（発行・入金確認）にアクセスした場合に拒否（403）されることを検証する（REQ-003）
- `accounting`ロールおよび`admin`ロールのみが請求書発行・入金確認操作を実行できることを検証する（REQ-064）
- Webルートでは403エラーページ、内部APIルートではJSON形式の403エラーレスポンスが返されることをそれぞれ検証する（api-endpoints.md「権限制御」）
**実装ファイル**: `tests/Feature/RoleBasedAccessControlTest.php`, `tests/Feature/Api/InternalApiAccessControlTest.php`

### 3. EDGEケースの網羅的テスト実装 🔵
**信頼性**: 🔵 *requirements.md Edgeケース（EDGE-001〜EDGE-011）より*

- EDGE-003: PDF生成に失敗した場合、エラーメッセージが表示され再試行を促す導線が提供されることを検証する
- EDGE-004: 同一受注に対する二重請求発行が防止されることを検証する
- EDGE-010: 在庫数が0の製品の受注登録ができない（警告表示）ことを検証する
- EDGE-011: 受注明細が0件の場合、受注確定ボタンが非活性になる（またはサーバー側でも確定処理が拒否される）ことを検証する
- 各EDGEケースについて、画面操作レベル（Featureテスト/ブラウザテスト）とAPI/サービスレベル（在庫チェックAPI・各Service層）の両方で検証する
**実装ファイル**: `tests/Feature/EdgeCaseStockTest.php`, `tests/Feature/EdgeCaseInvoiceTest.php`, `tests/Feature/EdgeCaseImportTest.php`, `tests/Feature/EdgeCasePdfGenerationTest.php`

## 統合テスト要件

### 統合テスト1: フルフロー（見積作成→PDF出力→受注確定→出荷指示→出荷完了→請求発行→入金CSV取込→ステータス確認）シナリオテスト 🔵
**信頼性**: 🔵 *dataflow.md全体フロー（見積→受注→出荷→請求→入金の一連シーケンス）より*
- **シナリオ**:
  1. salesユーザーが見積を作成し、見積PDFを出力する
  2. 見積を受注に変換し受注確定する（在庫引当が発生する）
  3. warehouseユーザーが出荷指示を作成し出荷完了とする（在庫が減算される）
  4. accountingユーザーが請求書を発行する（PDF生成含む）
  5. 振込データCSVを取り込み、入金消込を行う
  6. 受注・出荷・請求・入金それぞれのステータスが最終的に正しい状態（完了/消込済み等）になっていることを確認する
- **期待結果**: 一連の操作がエラーなく完了し、各ステップでのデータベース状態（在庫数、引当数、請求金額、入金消込状態）が整合していること

### 統合テスト2: 異常系シナリオ（在庫不足・二重請求・権限外アクセス・確定後編集制限） 🔵
**信頼性**: 🔵 *EDGE-001, EDGE-004, REQ-003, REQ-064、確定後ステータス遷移制約より*
- **シナリオ**:
  1. 在庫数を上回る数量で受注確定操作を行い、処理が中止され警告が表示されることを確認する（EDGE-001）
  2. 既に発行済みの受注に対して再度請求書発行操作を行い、二重発行が拒否されることを確認する（EDGE-004）
  3. warehouseユーザーで請求書操作画面・APIにアクセスし、403応答が返ることを確認する（REQ-003）
  4. ステータスが確定済み（confirmed以降）の受注に対する編集操作が制限される（編集不可または警告表示）ことを確認する
- **期待結果**: いずれの異常系操作も適切に拒否・警告され、データの不整合（在庫のマイナス、二重請求レコードの作成等）が発生しないこと

### 統合テスト3: 4役割によるアクセス制御マトリクステスト 🔵
**信頼性**: 🔵 *REQ-002, REQ-003, REQ-064、api-endpoints.md「権限制御」より*
- **シナリオ**:
  1. admin/sales/warehouse/accountingの各ロールでログインし、顧客・製品・見積・受注・出荷・請求書・レポート・内部APIの各機能へアクセスする
  2. 各ロールに許可された機能は正常にアクセスでき、許可されていない機能は403（画面はエラーページ、APIはJSON）が返されることを確認する
- **期待結果**: api-endpoints.md「権限制御」の役割別アクセス可能機能定義と実際の挙動が完全に一致すること

## 注意事項
- 各機能タスク（TASK-0005〜TASK-0015）が完了していることを前提に、機能間の連携・データ整合性を横断的に検証する。本タスクの開始前に依存タスクの単体・統合テストがすべて成功していることを確認すること
- E2Eテストはデータベースのトランザクション分離（`RefreshDatabase`等）を活用し、テスト間の状態汚染を防止すること
- PDF生成・CSV取込等、外部I/Oやファイルシステムに依存する処理はテスト用のフェイク/モックを活用し、テストの再現性と実行速度を確保すること
- 受け入れ基準テストケース（TC-001〜TC-040）の網羅状況はカバレッジ表として可視化し、抜け漏れがないことをチームでレビューできるようにすること
