## タスク概要
全銀協フォーマットの振込データCSVをアップロードし、請求書番号・金額により請求書と自動照合して入金ステータスを更新する機能を実装する（accounting/admin向け）。`PaymentImportService`を中心に、CSVのパース・照合ロジック・`payments`レコード作成（`source=csv_import`、`raw_csv_row`に元データ保存）・照合結果サマリー（成功件数・未照合件数）の表示までを実装する。dataflow.md「機能4: 振込データCSVインポート（入金照合）」のシーケンスに従い、`PaymentController → PaymentImportService → Repository → DB`の流れで処理を行う。

## 実装詳細

### 1. PaymentImportServiceクラス実装（全銀協フォーマットCSVのパース・請求書番号や金額による照合ロジック） 🔵
**信頼性**: 🔵 *REQ-063「システムは全銀協フォーマットの振込データCSVをインポートして入金ステータスを自動更新できなければならない」、architecture.md（PaymentImportService.php: 全銀協CSV取込）、dataflow.md「機能4」（S->>S: CSVをパースし振込データの行を抽出 / loop各振込データ行 / S->>R: 請求書番号・金額で請求書を検索）より*
- `PaymentImportService::importBankCsv(UploadedFile $file): PaymentImportResultData`: アップロードされたCSVファイルを全銀協フォーマット（固定長または区切り形式の振込明細レコード）に従ってパースし、各行から振込先名・金額・摘要（請求書番号を含む可能性のある文字列）等を抽出する
- 抽出した各行に対し、摘要欄等から請求書番号を特定し、`InvoiceRepository::findByInvoiceNumber()`で該当請求書を検索する。請求書番号が特定できない場合は金額による照合（同一金額の未入金請求書検索）にフォールバックする
- 処理結果は`PaymentImportResultData` DTO（data-types.phpに定義済み: `matchedCount`, `unmatchedCount`, `unmatchedItems`）として返却する
**実装ファイル**: `app/Services/PaymentImportService.php`, `app/Services/BankCsvParser.php`

### 2. PaymentController実装（インポート画面表示、CSVアップロード処理、照合結果表示） 🔵
**信頼性**: 🔵 *api-endpoints.md（GET /payments/import, POST /payments/import）、dataflow.md「機能4」（V->>C: POST /payments/import / C->>S: importBankCsv(file) / C-->>V: 結果表示）より*
- `index()`: `GET /payments/import` - 振込データCSVインポート画面（ファイルアップロードフォーム）を表示する。アクセス可能ロールはaccounting, admin
- `import(ImportPaymentCsvRequest $request)`: `POST /payments/import` - アップロードされたCSVファイルのバリデーション（形式・サイズ）後、`PaymentImportService::importBankCsv()`を呼び出し、結果サマリー（「N件成功、M件未照合」）と未照合明細を画面に表示する。アクセス可能ロールはaccounting, admin
- ロールごとのアクセス制御は`EnsureUserHasRole`ミドルウェアを利用する
**実装ファイル**: `app/Http/Controllers/PaymentController.php`, `app/Http/Requests/ImportPaymentCsvRequest.php`, `routes/web.php`

### 3. 照合できない件のスキップ処理・未照合件数の集計とレポート表示（EDGE-002） 🔵
**信頼性**: 🔵 *EDGE-002「振込データCSVインポート時に照合できない件があった場合、システムはスキップして未照合件数を報告しなければならない」、dataflow.md「機能4」（else照合失敗 / S->>S: 未照合リストに追加（EDGE-002））、data-types.php（PaymentImportResultData: unmatchedCount, unmatchedItems）より*
- `PaymentImportService`内で、請求書番号・金額のいずれでも照合できなかった行は処理をスキップし、`unmatchedItems`配列（`invoiceNumber`, `amount`, `reason`を含む連想配列）に追加する
- 全行処理後、`matchedCount`（成功件数）・`unmatchedCount`（未照合件数）・`unmatchedItems`（未照合明細）を集計し`PaymentImportResultData`として返す
- `PaymentController::import()`は結果サマリー「N件成功、M件未照合」を画面上部に表示し、未照合明細は理由（`reason`: 例「該当する請求書が見つかりません」「金額が一致しません」）とともに一覧表示する
**実装ファイル**: `app/Services/PaymentImportService.php`, `resources/views/payments/import_result.blade.php`

### 4. paymentsレコード作成（source=csv_import=2、raw_csv_rowに元データ保存） 🔵
**信頼性**: 🔵 *database-schema.sql（paymentsテーブル: source TINYINT（1=manual/2=csv_import）, raw_csv_row TEXT「全銀CSV取込時の元データ保持（監査用）」）、data-types.php（PaymentSource enum: MANUAL=1/CSV_IMPORT=2）より*
- 照合成功時、`PaymentRepository::create()`により`payments`レコードを作成する: `invoice_id`（照合された請求書ID）、`amount`（振込金額）、`paid_at`（振込日）、`source=PaymentSource::CSV_IMPORT`（コード値2）、`raw_csv_row`（CSVの元行データをそのまま文字列として保存、監査用）
- `raw_csv_row`への保存により、照合内容の事後検証・監査トレイルを確保する
**実装ファイル**: `app/Models/Payment.php`, `app/Repositories/Contracts/PaymentRepositoryInterface.php`, `app/Repositories/Eloquent/PaymentRepository.php`, `app/Services/PaymentImportService.php`

### 5. 入金額の累計に応じたpayment_status自動判定・更新（unpaid/partial/paid） 🔵
**信頼性**: 🔵 *REQ-062「システムは入金ステータス（未入金・部分入金・入金済み）を管理できなければならない」、dataflow.md「機能4」（S->>R: 入金ステータスを更新（部分入金/入金済み））、data-types.php（PaymentStatus enum: UNPAID=1/PARTIALLY_PAID=2/PAID=3）より*
- 照合成功した請求書に対し、`payments`テーブルの当該`invoice_id`に紐づく入金額合計（既存分＋今回分）を算出する
- 入金累計額と請求書の`total_amount`を比較し、`payment_status`を自動判定する: 累計額が0の場合は`UNPAID`（未入金）、`total_amount`未満の場合は`PARTIALLY_PAID`（部分入金）、`total_amount`以上の場合は`PAID`（入金済み）
- `InvoiceRepository::updatePaymentStatus()`（TASK-0012で実装）を呼び出し、`invoices.payment_status`を更新する
**実装ファイル**: `app/Services/PaymentImportService.php`, `app/Services/InvoiceService.php`

## 単体テスト要件

### テストケース1: 全銀協フォーマットCSVが正しくパースされる 🔵
**信頼性**: 🔵 *REQ-063「全銀協フォーマットの振込データCSVをインポート」、dataflow.md「機能4」（S->>S: CSVをパースし振込データの行を抽出）より*
- **Given**: 全銀協フォーマットに準拠した振込データCSVファイル（複数の振込明細行を含む）が用意されている
- **When**: `BankCsvParser::parse($file)`または`PaymentImportService::importBankCsv($file)`を呼び出す
- **Then**: 各行から振込先名・金額・振込日・摘要（請求書番号情報）等が正しく抽出され、構造化されたデータとして取得できること

### テストケース2: 金額が一致しない場合は未照合として記録されスキップされる（EDGE-002） 🔵
**信頼性**: 🔵 *EDGE-002「照合できない件があった場合、システムはスキップして未照合件数を報告しなければならない」より*
- **Given**: CSV内のある振込データ行の金額が、対応する請求書の`total_amount`（および入金累計の期待値）と一致しない、または該当する請求書が存在しない
- **When**: `PaymentImportService::importBankCsv($file)`を実行する
- **Then**: 当該行は`payments`レコードとして作成されずスキップされ、`PaymentImportResultData::unmatchedCount`がインクリメントされ、`unmatchedItems`に当該行の情報が含まれること

### テストケース3: 部分入金の場合payment_statusがpartial(2)になる 🔵
**信頼性**: 🔵 *REQ-062、data-types.php（PaymentStatus::PARTIALLY_PAID = 2）より*
- **Given**: `total_amount=100,000`の請求書（`payment_status=未入金`）が存在し、CSVから`amount=50,000`の振込データが照合される
- **When**: `PaymentImportService::importBankCsv($file)`を実行する
- **Then**: `payments`レコードが作成され（`source=csv_import`）、入金累計が`total_amount`未満であるため`invoices.payment_status`が`PARTIALLY_PAID`（コード値2）に更新されること

## 注意事項
- 全銀協フォーマットの仕様（固定長レコード形式・文字コード（多くの場合JIS/Shift-JIS）等）を確認したうえでパーサーを実装すること。文字コード変換が必要な場合は`mb_convert_encoding`等で適切に処理すること
- 照合ロジックは「請求書番号による照合」を優先し、特定できない場合のみ「金額による照合」にフォールバックする設計とすることで誤照合のリスクを低減すること
- `raw_csv_row`への元データ保存は監査要件として必須（database-schema.sqlコメントに明記）であるため、パース前の生データを欠落なく保存すること
- 入金ステータスの自動判定はTASK-0012で実装した`InvoiceService`の更新ロジックを再利用し、ロジックの重複実装を避けること
- 大量データのCSV取込によるメモリ枯渇を避けるため、ファイル読み込みはストリーミング処理（チャンク単位での読み込み）を検討すること
- ロールベースアクセス制御は`EnsureUserHasRole`ミドルウェアに委譲し、Controller内でのロール判定ロジックの重複を避けること
