# 要件定義・テスト作成 / 実装の分業フロー

要件定義・仕様作成・テストケース作成・完了チェックを自分（仕様担当）が行い、実装を別の担当者（実装担当）に依頼する場合の進め方をまとめたガイド。Tsumiki TDDの6フェーズを「仕様担当が担う範囲」「実装担当が担う範囲」に分割し、引き渡しポイントで何を渡す・確認するかを明確にする。

## 全体の役割分担

| フェーズ | コマンド | 担当 | 成果物 |
|---|---|---|---|
| 1. 要件定義 | `/tsumiki:tdd-requirements` | 仕様担当 | `{feature}-requirements.md` |
| 2. テストケース洗い出し | `/tsumiki:tdd-testcases` | 仕様担当 | `{feature}-testcases.md` |
| 3. Red（失敗テスト作成） | `/tsumiki:tdd-red` | 仕様担当 | `{feature}-red-phase.md`、失敗するテストファイル本体 |
| **── 引き渡しポイント①（仕様→実装） ──** | | | |
| 4. Green（最小実装） | `/tsumiki:tdd-green` | 実装担当 | `{feature}-green-phase.md`、実装コード |
| 5. Refactor（品質改善） | `/tsumiki:tdd-refactor` | 実装担当 | `{feature}-refactor-phase.md`、リファクタ後コード |
| **── 引き渡しポイント②（実装→仕様） ──** | | | |
| 6. 完了検証 | `/tsumiki:tdd-verify-complete` | 仕様担当 | `{feature}-memo.md`（最終記録）、タスクファイルの完了マーク |

ポイントは、**Redフェーズで作成する「失敗するテスト」が実行可能な仕様書として機能する**こと。テストコード自体に日本語コメント（テスト目的・期待される動作・信頼性レベル）が含まれているため、実装担当者は「何を満たせば正解か」をテストから直接読み取れる。

## 引き渡しポイント①: 仕様担当 → 実装担当

### 渡すもの
- `{feature}-requirements.md`（要件定義）
- `{feature}-testcases.md`（テストケース定義）
- `{feature}-red-phase.md`（作成したテストの一覧・期待される失敗・次フェーズへの要求事項）
- `{feature}-memo.md`（現状のフェーズ・関連ファイル一覧）
- 失敗状態のテストファイル本体（`tests/...`）

### 実装担当者に伝えること（チェックリスト）
- [ ] 通すべきテストファイル名と実行コマンド（例: `php artisan test --filter="XxxTest"`）
- [ ] テストコードの日本語コメントを読めば実装すべき仕様が分かる旨、および `{feature}-testcases.md` も併読してほしい旨
- [ ] **テストコード自体は変更・削除しない**こと（仕様の正はテスト側にあるため、実装側で合わせ込む）
- [ ] 実装の置き場所・大まかな方針（タスクファイルの「実装詳細」セクションを共有）
- [ ] Greenフェーズ完了の定義: 該当テストが全てグリーンになり、`{feature}-green-phase.md` に実装内容・方針・テスト結果を記録すること
- [ ] 既存テストが落ちた場合は、仕様変更とみなして良いか／要相談かの判断基準（基本は「仕様に照らして正しい方を採用し理由を記録する」）
- [ ] Refactorフェーズまで実装担当が行うか、Green止まりで仕様担当がレビューするかの分担確認

## 引き渡しポイント②: 実装担当 → 仕様担当（検証）

### 受け取るもの
- `{feature}-green-phase.md`（実装コード全文・実装方針・テスト結果・課題）
- `{feature}-refactor-phase.md`（改善内容、セキュリティ・パフォーマンスレビュー、最終コード）
- 実装コード本体（diffまたはファイル一覧）

### チェックすること（完了検証フェーズで実施）
- [ ] 該当範囲のテストを自分でも再実行し、全件グリーンであることを確認
- [ ] 全体テストスイートを実行し、スコープ外の失敗が新たに発生していないか確認
- [ ] `{feature}-testcases.md` の予定テストケースと実装済みテストの照合（要件網羅率の算出）
- [ ] 要件定義書（REQ-XXX, NFR-XXX）の各項目が実装・テストでカバーされているか
- [ ] コード品質（信頼性レベルコメントの有無、ファイルサイズ、モック使用の有無）
- [ ] セキュリティ・パフォーマンスレビューの内容に重大な懸念がないか
- [ ] 問題があれば `{feature}-memo.md` に記録し、後工程（追加実装・auto-debug等）に引き継ぐ

検証後、問題がなければ `{feature}-memo.md` を完了記録形式に更新し、元タスクファイルの完了条件チェックリストに完了マークを付ける。

## 運用上のヒント

- **フェーズ境界でのドキュメント更新を必須にする**: `{feature}-memo.md` は常に「現在のフェーズ」と「次フェーズへの要求事項」を含めて更新してもらうことで、引き継ぎ時の手戻りを減らせる。
- **テストは仕様、コードは実装の証**という役割分担を明示する: 実装担当者がテストの意図に疑問を持った場合は、テストを書き換えるのではなく仕様担当に確認するルールを徹底する。
- **Red phase記録の「次のフェーズへの要求事項」を厚めに書く**: ここが実質的な実装依頼書になるため、信頼性レベル（🔵🟡🔴）と合わせて「何が確定事項で何が推測か」を明記しておくと、実装担当者が独自判断で仕様を変えてしまうリスクを抑えられる。
- **小さく引き渡す**: 1タスク内でも機能単位でRed→引き渡し→Green→検証のサイクルを細かく回すと、認識ズレの早期発見につながる（TASK-0003では6ファイルに分割したことで管理しやすくなった）。
