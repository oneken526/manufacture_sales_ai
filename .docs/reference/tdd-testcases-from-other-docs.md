# タスクファイル以外の資料からテストケースを作成する方法

`docs/tasks/manufacture-sales-system/` 配下のタスクファイル以外でも、実装したい機能について記載した資料があれば、そこからテストケースを作成することは可能か、という疑問への回答メモ。

## 結論

可能。Tsumikiのワークフローはタスクファイル（`docs/tasks/.../TASK-XXXX.md`）を起点にしているが、実際にテストケース作成の材料として使われるのは「機能について書かれた資料」全般である。

## 1. 任意の資料から要件定義を起こす（`/tsumiki:tdd-requirements`）

- このコマンドは `docs/tasks/.../TASK-XXXX.md` を中心に読むが、タスクノート作成時（`tdd-tasknote`）には `docs/spec/*.md` や設計文書（`docs/design/...`）も探索対象になっている。
- 実装したい機能を説明した仕様書・設計メモ・ユーザーストーリーなどを `docs/spec/` 配下や該当する場所に置いておけば、それを根拠資料として要件定義書（`{feature_name}-requirements.md`）に反映できる。
- 既存のタスクファイルがない新規機能の場合でも、「要件名」「タスクID」「参照すべき資料のパス」を明示して依頼すれば、その資料をベースに要件定義から始めることが可能。

## 2. 要件定義からテストケースを洗い出す（`/tsumiki:tdd-testcases`）

- このフェーズは直前の要件定義書（および既存のテストケース）を読み込んでテストケースを設計する。
- 渡した資料の内容が要件定義書に正しく落とし込まれていれば、そのままテストケースにも反映される。

## 実務上のおすすめの進め方

1. 実装したい機能の資料（仕様書、画面イメージ、業務フロー説明、APIの入出力定義など）を用意する
2. その資料のパスと「この内容でTASK-XXXXのテストケースを作りたい」という意図を伝える
3. 必要に応じて簡単なタスクファイル（タイトル・概要・完了条件程度）を一緒に用意してもらえると、Tsumikiの定型フォーマットに沿いやすくスムーズ

## まとめ

タスクファイルが「唯一の入口」というわけではなく、機能を説明する資料があれば、それを要件定義の根拠として取り込み、そこからテストケースを派生させることができる。資料の形式（Markdown、PDF、画面のスクリーンショット等）も基本的に問わない。
