# TASK-0003 TDD開発フェーズ解説

TASK-0003（認証・権限基盤実装）で実行したTodoリストの各項目は、Tsumikiが定義するTDD開発の6フェーズに対応している。それぞれの内容を解説する。

## 1. 要件定義作成（tdd-requirements）

タスクファイル（TASK-0003.md）やEARS要件定義書（REQ-001〜005, NFR-010〜012）を読み込み、「何を実装すべきか」を機能要件・入出力仕様・制約条件・想定される使用例として明文化するフェーズ。各項目に信頼性レベル（🔵元資料に基づく／🟡妥当な推測／🔴根拠なき推測）を付け、後続フェーズが参照する「仕様の正」を作る。

## 2. テストケース洗い出し（tdd-testcases）

要件定義書をもとに、正常系・異常系・境界値の観点で網羅的なテストケースを設計するフェーズ。今回は「TC-N-01〜05（正常系5件）」「TC-A-01〜04（異常系4件）」「TC-B-01〜04（境界値4件）」の計13ケースを定義した。実装にまだ着手せず「何をテストするか」だけを先に固める点がポイント。

## 3. Redフェーズ（失敗テスト作成）

定義したテストケースに対応する実テストコードを書き、まだ実装が存在しないため失敗することを確認するフェーズ。「失敗する」こと自体がテストが正しく対象機能を検証できている証拠になる。今回は6ファイル・17テストメソッドを作成し、全件が期待通り失敗（`Target class [role] does not exist.`等）することを確認した。

## 4. Greenフェーズ（最小実装）

Redフェーズで失敗したテストを通すために必要最小限の実装を行うフェーズ。コードの美しさより「テストが通ること」を優先する。今回は`EnsureUserHasRole`ミドルウェア、`hasRole()`ヘルパー、ロール別リダイレクト、`is_active`チェック、Gate/Policy基盤などを実装し、新規17テスト全件成功・全体スイート58/58成功を確認した。

## 5. Refactorフェーズ（品質改善）

Greenフェーズで「動くようになったコード」を、機能を変えずに設計・可読性・保守性の観点で改善するフェーズ。今回はロール名⇔`UserRole` Enumの変換ロジックが2箇所に分散していた課題を`UserRole::routeKey()`/`fromRouteKey()`に一元化し、デッドコードも除去した。リファクタ前後でテスト結果に差異がないことを確認し、セキュリティ・パフォーマンスレビューも実施する。

## 6. 完了検証（verify-complete）

最後に、コードは一切修正せず、これまでの実装が「予定通り完成しているか」を検証・記録するフェーズ。具体的には、全テストの再実行による緑確認、予定テストケースと実装済みテストの照合、要件網羅率の算出、品質判定、そしてmemoファイル・元タスクファイル（完了マーク）の更新を行う。今回は要件網羅率100%・テスト成功率100%で「✅高品質（完全達成）」と判定した。

---

この6フェーズは「仕様を固める→テストで仕様を表現する→最小実装で通す→品質を上げる→完成を検証する」という一連の流れになっており、実装が要件から逸脱しないように各段階でドキュメント（requirements/testcases/red-phase/green-phase/refactor-phase/memo）に記録を残しながら進める設計になっている。
